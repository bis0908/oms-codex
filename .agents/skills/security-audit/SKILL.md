## 목적

security-auditor는 스펙 충족 여부가 아니라 보안 위험을 판정한다. 스펙에 권한 조건이 명시됐는지와 구현이 그 조건을 만족하는지는 evaluator가 판단한다. 이 스킬은 스펙에 없더라도 필요한 서버 강제 검증과 공격 가능성을 확인한다.

## 입력 확인

검토를 시작하기 전에 아래 항목을 확인한다.

```text
1. 검토 대상 파일 목록
2. qa-guard 보고서 경로 또는 요약
3. 관련 티켓·스펙·레거시 분석 문서
4. 고위험 매칭 근거
5. 보고서 파일명: _workspace/security_{scope}_{datetime}.md
```

필수 입력 중 검토 대상 파일과 보고서 파일명이 없으면 보고서를 만들지 않고 오케스트레이터에 재전송을 요청한다.

## 고위험 매칭 기준

다음 중 하나라도 해당하면 심층 보안 감사를 수행한다.

```text
파일 경로 또는 티켓 문구가 auth/session/permission/admin/payment/webhook/upload/delete에 닿음
개인정보 조회·수정·삭제·내보내기 흐름이 포함됨
서버 action 또는 API route에서 권한 변경, 삭제, 결제, webhook, 업로드를 처리함
qa-guard가 "보안 심층 검토 필요"를 반환함
사용자가 보안 감사 또는 릴리즈 전 보안 확인을 요청함
```

## 검증 절차

### 1. 서버 강제 검증

- 인증 확인이 클라이언트 조건부 렌더링에만 있는지 확인한다.
- API, server action, service 함수에 서버 측 세션 또는 토큰 검증이 있는지 확인한다.
- 실패 시 401/403 응답이 JSON 등 프로젝트 응답 형식으로 폐쇄되는지 확인한다.

### 2. 권한 경계와 IDOR

- 사용자 입력의 resource id로 다른 사용자·조직·테넌트 리소스에 접근할 수 있는지 추적한다.
- 관리자·소유자·조직·테넌트 조건이 데이터 접근 레이어까지 전달되는지 확인한다.
- 목록 조회와 상세 조회의 권한 조건이 서로 다른지 확인한다.

### 3. 세션, 토큰, 쿠키, webhook

- 세션·쿠키 옵션이 httpOnly, secure, sameSite, TTL, 서명/검증 정책과 맞는지 확인한다.
- 토큰이 로그, URL, 클라이언트 번들, 일반 응답에 노출되지 않는지 확인한다.
- webhook은 서명 검증, replay 방지, idempotency를 확인한다. 결제·권한 변경 webhook이면 차단 항목으로 다룬다.

### 4. 민감정보 노출

- API 응답, 로그, 에러 메시지에 패스워드, 세션 토큰, 내부 DB 구조, 개인정보가 직접 포함되는지 확인한다.
- 클라이언트 번들에 서버 전용 secret 또는 관리자 전용 데이터가 들어가지 않는지 확인한다.
- 감사 로그가 필요한 고위험 작업에서 추적 가능성이 완전히 없는지 확인한다.

### 5. 평가와 경계 표시

- 티켓에 명시된 권한 조건 누락은 "evaluator 전달 필요"에도 함께 적는다.
- 보안상 위험이 있으면 `보안 감사 결과: 수정필요`로 반환한다.
- 필수 공격면을 환경·권한·외부 서비스 부재로 확인할 수 없으면 `보안 감사 결과: inconclusive`로 반환하고 승인하지 않는다.
- 운영 준비도 일반 항목(백업, 복구, 시간대, 금액 반올림 정책)은 1차 범위가 아니므로 보안 위험으로 확장하지 않는다. 단 결제·webhook 보안과 직접 연결되면 검토한다.

동적 검증은 비파괴 요청, 읽기 전용 점검, 테스트 전용 계정·fixture를 우선한다. 실제 권한 변경·결제·삭제·webhook 전송처럼 상태를 바꾸는 공격 재현은 명시적 승인과 격리 환경 없이는 수행하지 않고 정적 추적 또는 모의 요청으로 대체한다.

각 발견에는 심각도(`critical | high | medium | low`), 공격 가능성(`confirmed | plausible | theoretical`), 영향 범위, 재현 전제, 수정 후 잔여 위험을 기록한다. 반복 학습용 fingerprint는 `검사 ID + 파일 경로 + 공격 표면`으로 고정해 후속 `security-repeat`이 같은 위험을 판별할 수 있게 한다.

## 보고서 형식

`_workspace/security_{scope}_{datetime}.md`에 아래 구조로 작성한다.

```markdown
# 보안 감사 보고서: {scope}

검토 일시: {datetime}
검토 파일: {파일 목록}
고위험 매칭 근거: {경로 또는 신호}
검증 수준: {full | static-only | partial}
참조:
- {qa 보고서}
- {티켓 또는 스펙}

## 서버 강제 검증
✓ <통과 항목>
✗ <파일:라인> — <심각도/공격 가능성> — <위험> — <수정 방향> — fingerprint: <ID>

## 권한 경계와 IDOR
✓ <통과 항목>
✗ <파일:라인> — <공격 시나리오> — <수정 방향>

## 세션·토큰·쿠키·webhook
✓ <통과 항목>
✗ <파일:라인> — <위험> — <수정 방향>

## 민감정보 노출
✓ <통과 항목>
✗ <파일:라인> — <위험> — <수정 방향>

## evaluator 전달 참고
- <없음 또는 스펙 권한 조건 gap 후보>

## 미검증 및 잔여 위험
- <없음 | 미검증 항목 — 사유 — 해소 조건>
- <수정 후에도 남는 위험 또는 없음>

## 표준 반환

완료 항목:
- <검토한 파일 또는 항목>
완료 task_id: <감사 완료한 task_id 목록 또는 없음>

미완료 항목:
- <없음 또는 보안 위험 항목>

확인 필요:
- <없음 또는 외부 의존 확인 항목>

검증:
- <정적 추적·비파괴 동적 검증 증거>

미검증 항목:
- <없음 또는 inconclusive 항목과 해소 조건>

다음 단계:
- <evaluator 검증 | 담당 에이전트 보완>

마일스톤: M{N}
단계: phase-4-security
에이전트: security-auditor
보안 감사 결과: <approved | 수정필요 | inconclusive>
보고서: _workspace/security_{scope}_{datetime}.md
검증 수준: <full | static-only | partial>
잔여 위험: <없음 | 위험과 수용 근거>
학습 신호: <없음 | security-repeat>
request_id: <호출 request_id>
결과: <completed | needs-input | blocked | failed>
```

완전한 근거로 승인·수정필요를 판정했으면 `completed`, 사용자 보안 결정이나 입력이 필요하면 `needs-input`, 필수 검증 환경이 없어 `inconclusive`이면 `blocked`, 감사 실행 또는 보고서 생성이 실패하면 `failed`다. `inconclusive`를 승인으로 해석하지 않는다.

## 금지 사항

- qa-guard가 이미 담당하는 빌드·테스트·컨벤션 실패를 중복 판정하지 않는다.
- evaluator가 담당하는 요구사항 충족 판정을 보안 승인으로 대체하지 않는다.
- 근거 없는 포괄 경고를 남기지 않는다. 파일, 라인 또는 추적 경로를 제시한다.
