## 재발 방지 학습 (작업 전 필독)

이 영역의 과거 QA 반복 실패 사례는 `docs/compound/qa/README.md` 인덱스에 유형별로 정리돼 있다(SKILL.md 본문의 `###` 능동 검증 절차는 그대로 유지. 해당 경로가 없으면 아직 누적된 학습이 없는 것이므로 건너뛰고 진행한다). 경로가 있으면 검증 시작 전 그 README를 읽고, 각 유형 파일 **첫 줄 load-when 술어**를 확인하여 현재 검증이 해당하는 사례 파일만 로드하라(전부 읽지 말 것). 사례를 이 SKILL.md에 누적하지 않는 이유는 스킬 로딩 컨텍스트를 상수 크기로 유지하기 위함이다 — compound-learner가 `docs/compound/`에만 기록한다(load-when 술어·인덱스 형식 규범은 `compound` 스킬의 `### Phase 2: docs/compound 작성` 참조).

## 검증 순서

0. 변경 파일 유형 분류 → 1. 보안 체크 → 2. 컨벤션 체크 → 3-0. 빌드 체크 → 3. 테스트 실행 → 4. 구조적 일관성 체크

## 책임 경계

qa-guard는 기술 품질과 빠른 정적 보안 탐지를 검증한다. 디자인 의미 판단과 심층 보안 감사는 하지 않는다.

| 담당 | 포함 |
|---|---|
| qa-guard | SQL placeholder, 하드코딩 시크릿, 응답 노출의 빠른 탐지, 프로젝트 언어·구조 컨벤션, 컴포넌트 위치, 테스트, DB 컬럼 흐름, 런타임 크래시 위험 |
| security-auditor | 서버 강제 인증/인가, IDOR, 세션·토큰·쿠키·webhook, 고위험 권한 변경, 민감정보 공격면 |
| design-reviewer | 색상·간격·라운드 토큰, hover/focus 시각 상태, 반응형, aria/alt 기초 |
| evaluator | 요구사항 충족, 사용자 플로우 연결, 제거 대상 기능 존재 여부 |

스타일 모듈은 런타임 크래시 또는 import 누락 여부만 확인한다. 색상·간격·토큰 미사용은 design-reviewer 보고서가 담당한다. 권한·인증·IDOR 같은 심층 보안 위험은 직접 판정하지 말고 `보안 심층 검토 필요` 신호로 넘긴다.

## 0. 변경 파일 유형별 집중 체크 (모든 마일스톤 필수)

보안·컨벤션 체크에 앞서 전달받은 파일 목록을 유형별로 분류한다. 사전에 알려진 버그 패턴이 아니라 **변경된 파일의 종류 자체**에서 추가 체크 항목을 파생한다. 어떤 작업이더라도 이 단계를 거치면 그 파일 유형에 맞는 체크가 자동으로 적용된다.

| 파일 유형 | 집중 체크 항목 |
|---|---|
| 데이터 접근 레이어 (쿼리/리포지터리) | SQL 전수 검사(섹션 1), SELECT 컬럼 목록이 실제 사용 필드를 모두 포함하는가, 파라미터 바인딩(placeholder) 누락 없는가, INSERT/UPDATE 페이로드 값의 타입이 DB 컬럼 타입과 일치하는가(예(프로젝트별): `enum('Y','N')` vs 정수 0/1 등) |
| API 엔드포인트 핸들러 | 세션 가드 호출 위치·순서, 응답 포맷 일관성(예(프로젝트별): `{ data, error }`), HTTP 메서드 export 명칭 일치 |
| 화면/페이지 컴포넌트 | 클라이언트 전용 코드 최소화(서버 우선 렌더링 원칙), 데이터 페칭 await 누락, 조건부 렌더링 양 분기 존재 |
| 재사용 컴포넌트 | prop 전달 체인 끊김, 클라이언트 전용 hook을 서버 컴포넌트에서 혼용, 필수 prop 누락 |
| 스타일 모듈 | 스타일 파일 존재, import 경로, 클래스명 오타로 인한 런타임 누락 위험 |
| DB 마이그레이션 | `ADD COLUMN` + 백필 `UPDATE` 쌍 존재, `NOT NULL` 컬럼에 `DEFAULT` 누락, 기존 데이터 무결성 |
| 공용 유틸/라이브러리 모듈 | 단일 책임 원칙 유지, 외부 의존성 최소화, 에러 전파 방식 일관성 |

**이 분류는 섹션 1~4의 실행 순서를 바꾸지 않는다.** 해당 유형의 파일이 포함됐을 때 그 섹션을 더 집중적으로 실행하라는 가중치다.

## 1. 빠른 정적 보안 체크

### SQL Injection 탐지

아래 패턴을 Grep으로 탐지한다.

```
탐지 패턴:
- `\$\{.*\}` in SQL strings (템플릿 리터럴 보간)
- `"SELECT.*\$` (문자열 연결 보간)
- `.format(` in SQL context

통과 기준:
- 모든 파라미터가 ? placeholder 사용
- 동적 식별자에 화이트리스트 적용 확인
```

### 하드코딩 자격증명 탐지

```
탐지 패턴:
- password\s*=\s*['"][^'"]+['"]  (소스 내 패스워드 리터럴)
- host\s*=\s*['"][0-9.]+['"]     (IP 직접 기재)
- secret\s*=\s*['"][A-Za-z0-9]+  (시크릿 리터럴)

통과 기준: 모든 자격증명이 환경변수 참조
```

### 응답 노출 탐지

API Route 응답에 세션 토큰, 패스워드, DB 내부 구조가 포함되지 않는지 확인한다.

### 보안 심층 검토 신호

아래 조건 중 하나라도 발견하면 실패 여부와 별개로 보고서와 표준 반환에 `보안 심층 검토 필요`를 기재한다. qa-guard는 이 항목을 직접 심층 감사하지 않는다.

```text
보안 심층 검토 필요 조건:
- 변경 파일 또는 티켓 문구가 auth/session/permission/admin/payment/webhook/upload/delete에 닿음
- 개인정보 조회·수정·삭제·내보내기 흐름 포함
- 서버 action 또는 API route에서 권한 변경, 삭제, 결제, webhook, 업로드 처리
- 클라이언트 권한 체크만 보이고 서버 강제 검증 여부가 불명확함
```

보고 형식:

```text
보안 심층 검토 필요: <없음 | 파일 목록과 근거>
```

## 2. 컨벤션 체크

### 언어·파일 형식 컨벤션

프로젝트 AGENTS.md가 정한 언어·파일 형식 규칙(허용/금지 확장자, 타입 시스템 사용 여부 등) 위반을 탐지한다.

```
금지 패턴 예(프로젝트별): 프로젝트가 금지한 파일 확장자·설정 파일·import 구문 존재
```

### 의존성 정책 컨벤션

프로젝트 AGENTS.md가 금지한 라이브러리·도구(예(프로젝트별): 특정 ORM/스키마 빌더 등)의 import를 탐지한다.

```
금지 패턴 예(프로젝트별): import.*<금지 라이브러리명>
```

### 컴포넌트 위치 규칙

프로젝트 AGENTS.md가 정한 영역/그룹 간 import 경계를 위반하지 않는지 확인한다.

```
위반 패턴 예(프로젝트별): 한 영역(그룹)에서 다른 영역 전용 컴포넌트를 직접 import
```

### UI 컴포넌트 화이트리스트 초과

프로젝트가 외부 UI 컴포넌트 라이브러리에 대해 허용 목록을 정한 경우, 목록 외 컴포넌트 import 감지 시 오케스트레이터 승인 여부를 확인한다(허용 목록은 프로젝트 AGENTS.md가 정의).

## 3-0. 빌드 체크

테스트 실행 전에 가장 싼 기계 신호를 먼저 확보한다(import 오타·문법 오류·컴파일 실패는 Grep·LLM 판정으로는 잡히지 않으며, 테스트 없는 마일스톤에서는 이 단계가 유일한 그라운드 트루스다).

- 프로젝트 AGENTS.md에 빌드/컴파일/타입체크 명령이 정의돼 있으면 **1회 실행**하고 종료 코드를 기재한다(구체 명령은 프로젝트 AGENTS.md가 정의 — 컨벤션 체크와 동일한 프로젝트 위임 패턴).
- 범위는 1회 컴파일/타입체크 수준으로 한정한다. dev 서버 기동·watch 모드·E2E 실행 같은 상주·장시간 작업은 이 단계의 대상이 아니다.
- 실패(종료 코드 0이 아님) 시 ✗ 처리하고 오류 출력 요약과 함께 조치 필요 목록에 올린다. 이때 섹션 3 테스트 실행은 "빌드 실패로 보류"로 표기하고 건너뛴다(같은 원인의 중복 실패 방지). 섹션 4 정적 체크는 계속 수행한다.
- 명령 미정의 시 이 단계를 건너뛰되 보고서에 "빌드 체크: 미정의 — skip"으로 명시한다(조용한 생략 금지 — 건너뜀은 명시가 전제다).

결과 기재 위치: 보고서 `## 테스트 결과` 섹션 **첫 줄**에 `빌드: {명령} — 종료 코드 {N}` (미정의 시 `빌드: 미정의 — skip`) 1줄로 기재한다. 보고서 섹션 구조 자체는 변경하지 않는다.

## 3. 테스트 실행

data-layer가 Red→Green 과정에서 작성한 테스트 파일을 실행한다. 이 단계에서 새 테스트를 작성하지 않는다.

### 실행 방법

프로젝트 테스트 러너로 전체 또는 특정 파일을 실행한다.

```bash
# 전체 실행
<프로젝트 테스트 명령>

# 특정 파일만
<프로젝트 테스트 명령> {파일경로}
```

### 실행 결과 처리

| 결과 | 처리 |
|---|---|
| 전체 통과 | 보고서에 "테스트 통과: N개" 기재 |
| 일부 실패 | 실패 케이스 목록 + 사유를 보고서에 기재, 오케스트레이터에 담당 에이전트 보완을 요청 |
| 테스트 파일 없음 | "테스트 없음 — 대상 변경에 테스트 부재"로 표기하고 필요성을 판정한다. |
| 광역 회귀 다수 실패 (대상 외) | 선재결함 분리 진단 후 비차단 메모 — docs/compound/qa/regression-isolation.md 참조 |
| 환경·권한·서비스 문제로 실행 불가 | `QA 검증 결과: inconclusive`, 검증 부채와 해소 조건을 기록하고 승인하지 않음 |

**증거 동봉 (의무)**: 어느 결과든 보고서 `## 테스트 결과` 섹션에 ① 실제 실행한 테스트 명령과 ② 러너 요약 라인 원문 인용(예: `Tests: 12 passed, 12 total`)을 기재한다. 테스트를 실행하지 않았으면(테스트 없음·DB 미연결 등) 명령·인용 대신 미실행 사유 1줄을 기재한다(검증자도 LLM이다 — 실행 증거가 없는 "통과: N"은 실제 실행과 산출물로 구분되지 않으며, 인용이 없으면 오케스트레이터가 보고서 재전송을 요구한다).

### DB 테스트 원칙

- 프로젝트의 격리 test DB, test schema, transaction fixture 정책을 우선한다. 공유 dev DB의 기존 데이터에 의존하지 않는다.
- 테스트 목적과 프로젝트 정책이 허용하면 mock도 사용할 수 있다. DB 계약·query 통합 검증이 필요한 항목은 격리 DB로 별도 확인하고 mock 통과만으로 대체하지 않는다.
- 운영 DB는 절대 사용하지 않는다.
- 격리 test DB가 준비되지 않은 경우 정적 코드 검토만 수행하고 "test DB 미연결 — 런타임 검증 보류"로 명시한다. DB 검증이 필수 범위면 `QA 검증 결과: inconclusive`이며 통과로 계산하지 않는다.

## 4. 구조적 일관성 체크

DB 마이그레이션·신규 파라미터 컬럼 도입·컴포넌트 분리가 포함된 마일스톤(실데이터 단계 이후)에서 실행한다.

### DB 컬럼 흐름 추적

신규 컬럼을 참조하는 코드가 있으면 4단계 체인을 검증한다.

```
체인 검증 순서:
1. DB 마이그레이션 — ADD COLUMN {컬럼명} 존재 여부
2. 데이터 접근 레이어 — SELECT 목록에 {컬럼명} 포함 여부
3. 페이지/엔드포인트 핸들러 — 구조분해 또는 prop 전달 경로에 포함 여부
4. 렌더링 조건문 — {컬럼명} 값을 실제 렌더링 분기에 사용 여부

체인 중 빠진 단계 → "컬럼 흐름 끊김" 처리 (✗ 누락)
```

### 레거시 폴백 잔존 탐지

기존 컬럼을 대체하는 신규 파라미터 컬럼 도입 후에도 구 컬럼 기반 분기가 제거되지 않았는지 Grep으로 탐지한다.

```
탐지 패턴 예(프로젝트별): 신규 컬럼으로 교체했어야 할 구 컬럼 기반 조건 분기가 코드에 잔존
  (예: 구 분류 컬럼 비교가 신규 파라미터 컬럼 비교로 교체되지 않은 경우)

허용 예외:
- 데이터 접근 레이어 내 구 컬럼 직접 SELECT (마이그레이션 호환용)
- 레이블 표시 전용 참조 (구 컬럼 값으로 표시 문자열만 결정하는 경우)
```

탐지 시 처리: 교체 완료 여부를 확인하고, 미완료이면 보고서에 "폴백 잔존" 경고로 명시.

### 공개 계약 정적 대조

변경된 UI 컴포넌트, 데이터 쓰기 경로, 조건 분기, API handler는 프로젝트의 실제 타입·schema·컴포넌트·응답 계약과 대조한다. 특정 과거 버그의 증상 문자열이나 구현 예시를 상시 규칙으로 사용하지 않는다. 반복 사례가 있으면 `docs/compound/qa/`의 load-when 조건에 맞는 사례만 로드한다.

- 호출자와 피호출자의 인자·콜백·반환 계약이 일치하는지 확인한다.
- DB 컬럼명·타입·alias와 코드 소비 지점이 연결되는지 확인한다.
- 조건 분기의 생략이 요구사항상 의도인지 확인한다.
- endpoint의 성공·실패 응답 형식이 소비자 계약과 일치하는지 확인한다.
- 근거가 없으면 통과로 추정하지 않고 미확인 또는 수정필요로 분류한다.

## 보고서 형식

`_workspace/qa_{feature}_{datetime}.md` 에 아래 구조로 작성한다. 오케스트레이터 봉투가 보고서 파일명을 별도 지정하면(예: 증분 재게이팅 `qa_regate_*`, Phase R `qa_refactor_*`, 형식 재전송 시 직전 보고서와 동일 파일명 덮어쓰기) 기본 명명 대신 그 지정을 따른다(절대 라운드 상한의 파일 개수 기계 카운트 오염 방지).

머리말 `반복 횟수` 기입 규칙(외부 기억 — compaction 후 보완 루프 회차 복원용): 같은 feature의 직전 `_workspace/qa_*` 보고서가 있으면 그 머리말의 N에 +1, 없으면 `1회차`로 기재한다. `직전 미해결` 에는 그 직전 보고서의 ✗ 항목을 체크 섹션명 + 파일 경로(라인 번호 제외) 키로 옮겨 적는다 — 직전 보고서가 없으면 `없음`.

```markdown
# QA 보고서: {feature}

검증 일시: {datetime}
검증 파일: {파일 목록}
반복 횟수: {N}회차 / 직전 미해결: {목록}
검증 수준: {full | static-only | partial}

## 보안
✓ SQL Placeholder 사용 확인
✗ 파일명:라인 — 사유 — 수정 방법

## 컨벤션
✓ 언어·파일 형식 컨벤션 준수
✗ 파일명:라인 — 사유

## 테스트 결과
빌드: {명령} — 종료 코드 {N} (미정의 시 `빌드: 미정의 — skip`)
실행 명령: `{실제 실행한 테스트 명령}`
종료 코드: {N | 미실행}
러너 출력 인용: "{러너 요약 라인 원문 — 예: Tests: 12 passed, 12 total}"
통과: N / 실패: N
✗ 실패 테스트명 — 사유
(미실행 시: 실행 명령·인용 두 줄 대신 미실행 사유를 한 줄로 기록한다.)

## 구조적 일관성 (해당 마일스톤만)
✓ DB 컬럼 흐름 — 마이그레이션 → 데이터 접근 레이어 → 컴포넌트 연결 확인
✗ 파일명:라인 — 흐름 끊김 위치 — 수정 방법
⚠ 레거시 폴백 잔존 — 파일명:라인 — 교체 여부
✓ 조건부 렌더링 대칭 — 상태 플래그/파라미터 컬럼 양 분기 확인

## 조치 필요
(담당 에이전트별 수정 요청 목록)

## 보안 심층 검토 필요
<없음 | 파일 목록과 근거>

## 검증 부채
- <없음 | 미검증 항목 — 사유 — 해소 조건>
```

## 반환 계약

공통 키는 `orchestrate/references/agent-contract.md`를 사용한다. 역할별로 `보고서`, `QA 검증 결과`, `호출 ID`, `검증 수준`, `검증 부채`, `보안 심층 검토 필요`를 추가한다.

필수 검증을 모두 실행해 판정했으면 `completed`, 사용자·요구사항 확인이 필요하면 `needs-input`, 환경 때문에 필수 검증을 실행하지 못한 `inconclusive`는 `blocked`, 검증 명령이나 보고서 생성 자체가 실패하면 `failed`다. `inconclusive`를 `approved`로 반환하지 않는다.

## 검증 제외

아래 항목은 실패로 판정하지 않고 담당 검증 결과를 참조한다.

- 디자인 토큰 미사용
- 색상·간격·라운드 불일치
- hover/focus 시각 상태
- aria-label 또는 img alt 같은 접근성 기초 항목
- 요구사항 누락 또는 초과 구현 여부
- 서버 강제 인증/인가, IDOR, 세션·토큰·쿠키·webhook, 민감정보 공격면 심층 판단
