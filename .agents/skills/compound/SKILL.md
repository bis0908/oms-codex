## 목적

같은 실수가 반복되지 않도록 매 수정 후 예방 패턴을 `docs/compound/{카테고리}/` 에 유형별로 기록한다.
`docs/compound/` 디렉토리가 없으면 첫 학습 시 생성한다.
자가개선 경로: `bug-fixer/evaluator/qa-guard/design-reviewer` → `compound-learner` → `docs/compound/` 갱신 → 다음 작업 시 오케스트레이터 푸시 + SKILL.md 포인터 풀로 해당 사례 로딩.

**SKILL.md 직접 수정 금지.** 과거에는 대상 SKILL.md 의 "자주 발견되는 gap 패턴" 섹션에 누적했으나, 스킬 로딩 컨텍스트 무한 증가 문제로 폐기됐다. 출력은 `docs/compound/` 에만 쓴다(이 `## 목적` 절이 곧 배경이다). 소비 메커니즘은 `orchestrate` 스킬의 카테고리 README 주입 표 참조.

### 정리 트랙 (`compound-curate` 트리거)

학습이 아니라 **누적분 정리**다. 매핑표를 쓰지 않는다(대상 파일이 이미 카테고리에 있다). 입력: curate-needed 플래그의 대상 `{유형}.md` 목록(온디맨드 시 카테고리 전 유형 파일).

목적: 무손실로 로드 구간(원칙+활성 상단)을 상수에 수렴시킨다. 바이트 감소가 목표가 아니다. SKILL.md 무수정.

**2단계 절차:**

a. **분석**: 대상 `{유형}.md` 를 읽어 ① 중복/근접중복 사례 군집화, ② 한 사례가 다른 사례를 일반화·포괄하는 관계(supersede 관계)를 식별한다. ③ **자매 군집화 — 클래스별 그룹**: `자매({기준})` 토큰을 가진 사례를 `{기준}` 별로 묶어 같은 버그 클래스를 식별한다. 한 클래스가 4~5개 이름으로 호칭될 수 있으므로(앵커 키 불안정), 평면 `grep -c '자매('` 가 아니라 **기준별 그룹 카운트**(`grep -oE '자매\([^)]*\)' {유형}.md | sort | uniq -c`)로 센다 — 평면 카운트는 서로 다른 클래스의 자매를 한 덩어리로 합산해 과대·과소 판정을 부른다. **정리 전 사례 수를 기계 카운트로 포착한다**: 사례 본문 표준 형식(아래 "사례 본문 형식")은 모든 사례가 말미에 `grep 키워드:` 토큰을 달고 승격 원칙·`→ 승격 원칙:` 포인터·헤더·load-when 술어는 달지 않으므로, `grep -c 'grep 키워드:' {유형}.md`(또는 동등 기계 카운트)로 정리 전 사례 수를 도구로 얻어 보관한다. 사람/LLM 셈으로 대체하지 않는다.

b. **curate-write (무손실)**:
   - 통합 가능한 규칙을 파일 상단 `## 원칙 (승격)` 섹션에 1~N줄로 승격한다(섹션 없으면 신설). 첫 줄 `> load-when:` 술어는 보존한다. **앵커 정규화 (자매 클래스)**: a③ 에서 식별한 각 자매 클래스에 대해 클래스 정규 앵커(대표명) 1개를 이 `## 원칙 (승격)` 요지에 고정하고, 이후 그 클래스의 자매 토큰·Phase 1 `재발/자매 대조` 3분기는 모두 이 앵커를 `{기준 제목}` 으로 인용한다 — 한 클래스가 여러 이름으로 호칭돼 자매 카운트가 과소집계되는 것(size-on-read 자매 트리거 미발동)을 막는 선행 조건이다.
   - 피지그/중복 구 사례 본문은 **삭제하지 않고** 파일 하단 `## _superseded (로드 불필요)` 섹션으로 이동하고, 각 항목 끝에 `→ 승격 원칙: {요지}` 포인터 1줄을 부착한다.
   - 원칙으로 흡수되지 않은 활성 사례는 `## 활성 사례` 섹션에 잔존시킨다.
   - 카테고리 `README.md` 인덱스 행을 재동기화: "다루는 사례"는 활성+원칙 기준으로 타이트닝한다. **grep 키워드 합집합 열은 README 에 두지 않는다(환원 — split 이 아니라 relocate)** — 검색 정본은 README 사본이 아니라 본문 `grep 키워드:` 토큰이며(`## 사례 본문 형식`), README 행에는 검색 레시피 포인터 `grep -rlF '<키워드>' docs/compound/{카테고리}/` 한 줄만 남긴다. 이 레시피는 카테고리 디렉토리 전체를 훑으므로 `## 활성 사례`·`## _superseded` 양 섹션까지 도달해 검색 커버리지가 생존한다(자매는 활성 섹션에 in-place 누적, _superseded 만 하단 섹션으로 격리 — 둘 다 같은 파일 안이고 레시피는 디렉토리 재귀). README 에서 키워드를 제거하기 전 각 키워드가 그 `{유형}.md` 본문에 `grep -lF '<키워드>' {유형}.md` 로 HIT 하는지 확인하고, **MISS 면 그 키워드를 본문의 해당(또는 가장 가까운) 사례 `grep 키워드:` 줄에 먼저 append(append-only) 한 뒤** README 에서 제거한다(검색 정본을 본문으로 옮긴 뒤에만 README 사본을 버린다 — fails-closed). 본문 append 없이 제거하면 검색 커버리지를 잃으므로 1건이라도 금지. (안전성 출처는 "키워드가 순수 중복"이 아니라 이 `grep -lF` MISS→append→제거 순서다 — README-only 키워드가 1건이라도 존재할 수 있으므로 무조건 제거가 아니라 fails-closed 로 보장한다.)

c. **검증 (무손실 assertion — 기계 카운트)**: 사례 본문 총량 보존을 LLM 셈이 아니라 도구로 단언한다. curate-write 직후 동일 패턴으로 정리 후 사례 수를 다시 센다: `grep -c 'grep 키워드:' {유형}.md`(활성 사례 + _superseded 사례 합계 — 두 섹션 모두 사례 본문에 `grep 키워드:` 토큰이 남아 있고 승격 원칙·`→ 승격 원칙:` 포인터는 토큰이 없어 카운트에서 빠진다). 무손실 공식 `(정리 후 grep -c) == (a 단계에서 보관한 정리 전 grep -c)` 가 성립하는지 단언한다(작성자-검증자 분리 — 같은 에이전트라도 무손실 판정을 도구값 대조로 강제해 거짓 통과를 도구 1회로 차단). 1건이라도 어긋나면 미완료로 보고하고 종료한다(원복 가능 상태 유지). 두 카운트 모두 반드시 도구값이어야 한다 — 한쪽이라도 사람/LLM 셈이면 거짓 통과 구멍이 다시 열린다. **검색 커버리지 단언 (환원 전용, file-level — `grep -c` 와 직교)**: b 의 README 환원에서 제거 대상이 된 키워드 전부가 그 `{유형}.md` 본문에 `grep -lF '<키워드>' {유형}.md` 로 HIT 하는지 별도 단언한다(per-case 매핑이 아니라 file-level — 정규화된 키워드가 어느 사례 줄에 붙었든 그 파일 본문에 있으면 통과). 1건이라도 MISS 면 미완료로 보고하고 종료한다(검색 커버리지 손실 금지 — 원복 가능 상태 유지). 이 단언은 무손실 `grep -c` 단언과 **직교**한다: `grep -c 'grep 키워드:'` 는 README 를 만지지 않아 키워드가 README 에만 있다가 사라지는 환원 누락을 잡지 못하고, MISS-append 는 기존 사례의 `grep 키워드:` 줄에 토큰을 덧붙일 뿐 새 사례 bullet 을 만들지 않으므로 `grep -c` 불변식도 보존된다.

**불변식**: 어떤 사례 본문도 삭제하지 않는다. SKILL.md 는 어떤 경우에도 열거나 수정하지 않는다.

## Gap 유형 → docs/compound 카테고리 매핑표

학습 결과는 그것을 **소비할 에이전트의 카테고리** 디렉토리에 기록한다.

| gap / 버그 유형 | 카테고리 디렉토리 | 보완 트랙 |
|---|---|---|
| 프론트 조건부 분기·폼 바인딩·라우팅·UI 토큰·렌더링 버그 | `docs/compound/frontend/` | bug-fix, qa-repeat, design-repeat |
| SQL 오류·placeholder·status 필터·페이지네이션·JOIN·컬럼 흐름 | `docs/compound/data-layer/` | bug-fix, eval-gap, qa-repeat |
| 원본 스펙 섹션·플로우·실데이터 연결 누락, 제거/초과 판정 | `docs/compound/evaluation/` | eval-gap |
| QA 검증 절차·선재결함 분리·스모크 누락 | `docs/compound/qa/` | qa-repeat |
| CSS 토큰·hover/focus·반응형·접근성·브랜드 컬러 | `docs/compound/design/` | design-repeat |
| 레거시 소스 분석 추출 누락(섹션 인벤토리·URL 혼용·리다이렉트 래퍼·동반 스키마) | `docs/compound/legacy/` | eval-gap |
| 버그 진단 방법론(계층 진단·실DB 데이터 분기·원본 충실 예외) | `docs/compound/bugfix/` | bug-fix |
| 테스트 계약 고정 오류(placeholder 계약·mock 누락) | `docs/compound/tdd/` | qa-repeat |
| 하네스/오케스트레이터 메타 결함(게이팅 화이트리스트·라우팅 휴리스틱·권한 경계 등 하네스 운영 결정 사항) | `docs/compound/harness/` | bug-fix (메타 확장) |

매핑표에 없는 유형은 가장 가까운 카테고리를 선택하되, 두 카테고리에 동등하게 걸치면 "매핑 모호 — 수동 검토 필요"를 반환하고 종료한다. 원인 자체를 판별할 수 없으면 "매핑 불가 — 수동 검토 필요"를 반환한다.

**권한 경계 — 신 카테고리 신설은 compound-learner 권한 밖.** 매핑표에 없는 새 카테고리가 필요한 사례(예: 하네스 메타 결함)는 오케스트레이터(메인)가 직접 `docs/compound/{새카테고리}/` 디렉토리·README.md·유형 파일을 신설하고, 본 매핑표에 1행을 직접 추가한다. compound-learner에 작업 봉투로 "권한 부여" 우회 위임 금지 — 권한 경계가 모호해지고 단순 파일 생성 작업에 별도 Agent 컨텍스트 비용이 발생한다. compound-learner는 기존 매핑된 카테고리에 사례를 적재하는 작업만 수행한다.

## 2단계 실행

### Phase 1: 분석

1. 보고서 읽기: bug-fixer 반환 메시지 전문, `_workspace/eval_*.md`, `_workspace/qa_*.md`, `_workspace/design_review_*.md` 중 트리거에 맞는 입력을 읽는다.
2. 카테고리 식별: 위 매핑표에서 gap/버그 유형 대조 → `docs/compound/{카테고리}/` 확정.
3. 해당 카테고리 `README.md` 인덱스를 읽어, 추가할 사례가 **기존 유형 파일에 속하는지** 새 유형이 필요한지 판단한다.
   - **재발/자매 대조 (기존 유형 파일에 속하는 경우만) — 3분기**: 그 유형 파일의 기존 사례 본문과 새 사례를 대조하여 증상·원인·표면(발현 레이어)을 비교하고 아래 셋 중 하나로 분류한다. **보수 기본값: 불확실하면 ③ 신규로 둔다** — 자매·재발 과대표기는 거짓 군집 → 잘못된 승격 원칙 → 하류 오염을 부른다(0-사례 골격에선 흔들림이 잠복하나 실사례 누적 시 자매↔신규 비결정성으로 표면화될 수 있다).
     - **① 재발 (같은 증상·원인·같은 표면)**: 이미 적재된 사례다(= SKILL.md 포인터 풀·오케스트레이터 푸시로 주입된 학습이 소비되지 못했다는 가장 강한 신호). Phase 2 append 시 새 사례 제목 앞에 `재발({기존 사례 제목},{N}회차)` 표기를 붙이고(N은 원본 최초 적재를 1회차로 센 누적 발생 횟수 — 첫 재발 2회차, 이후 매 재발마다 +1), 반환 `확인 필요`에 `소비 경로 점검 필요 — {유형}.md 재발 ({N}회차)` 를 1줄 남긴다.
     - **② 자매 (같은 버그 클래스·다른 표면/레이어)**: 같은 뿌리가 다른 표면으로 발현한 경우다(예: 같은 크로스커팅 결함이 한 번은 테이블 정렬 버그, 한 번은 표시 포맷 버그로). 새 사례 제목 앞에 `자매({기준 제목})` 토큰을 붙인다 — `{기준 제목}` 은 이 클래스의 **정규 앵커**다(curate step-b 가 `## 원칙 (승격)` 요지에 클래스 대표명을 1회 고정했으면 그것을 인용하고, 아직 승격 전이면 클래스 최초 사례 제목을 기준으로 삼는다 — curate 시 앵커로 수렴, `### 정리 트랙` 참조). 자매는 정상 누적이므로 `확인 필요`를 남기지 않는다 — 소비는 grep 이다(한 기준의 자매가 임계만큼 쌓이면 `orchestrate` 의 size-on-read 가 curate-needed 로 잡는다).
     - **③ 신규 (공유 버그 클래스 없음)**: 평문 사례로 그대로 append 한다.
     - **재발·자매 어느 경우든 기존 사례를 삭제·재구성하지 않는다(append-only 불변식 유지) — 이 대조는 재발/자매 감지기이지 dedup 삭제기가 아니다.** 기존 사례는 그대로 두고 새 사례만 표기해 추가한다.

### Phase 2: docs/compound 작성 (2-write 원자 단위)

1. **유형 파일 본문 append**:
   - 기존 유형에 속하면 `docs/compound/{카테고리}/{유형}.md` 끝에 사례 1건 추가. 파일 첫 줄 `> load-when:` 술어가 새 사례를 포괄하지 못하면 술어를 확장한다.
   - 새 유형이면 `docs/compound/{카테고리}/{새유형}.md` 를 생성하고 **첫 줄에 `> load-when: <술어 한 줄>`** 을 기재한 뒤 사례를 작성한다(사례 본문 레이아웃은 이 문서의 `## 사례 본문 형식`, 정리 후 `## 원칙 (승격)`·`## 활성 사례`·`## _superseded (로드 불필요)` 구조는 이 문서 `### 정리 트랙` 절 참조).
2. **카테고리 README 인덱스 갱신**:
   - 새 유형 파일을 만들었으면 `docs/compound/{카테고리}/README.md` 표에 1행 추가한다. 행에는 "다루는 사례"(routing 요약)와 검색 레시피 포인터 `grep -rlF '<키워드>' docs/compound/{카테고리}/` 를 두되, **grep 키워드 합집합 열은 두지 않는다(환원 불변식)** — 새 사례의 검색 정본은 본문 `grep 키워드:` 토큰이다(`## 사례 본문 형식`).
   - 기존 유형에 추가했으면 해당 행의 "다루는 사례"가 새 사례를 포괄하도록 갱신(필요 시)한다. **README 행에 grep 키워드를 재적재하지 않는다(환원 불변식)** — append 트랙이 키워드를 README 에 다시 쓰면 매 사례 추가마다 핫패스 README(오케스트레이터 작업 봉투에 버그 작업마다 통째 주입)가 다시 비대해져 curate 의 환원이 무효화된다. 새 사례의 검색 커버리지는 본문 `grep 키워드:` 토큰이 보유한다.
3. **검증**: 두 파일이 모두 갱신됐는지 확인. 하나만 갱신되면 미완료로 보고.

SKILL.md 파일은 어떤 경우에도 열거나 수정하지 않는다.

## 사례 본문 형식

아래 표준 형식을 따른다.

```
- **{유형 제목} ({M{N} 또는 버그 식별자} 사례)**: {무엇을 놓쳤는가/증상} — {어떻게 예방/판정하는가}. grep 키워드: `키워드1`, `키워드2`.
```

- 학습 4트랙은 기존 사례를 삭제·재구성하지 않고 항목만 추가한다(append-only). `compound-curate` 만 _superseded 격리/원칙 승격을 수행하되 본문은 무손실로 보존한다.
- 본문은 요약하지 말고 재발 방지에 필요한 절차를 그대로 기술한다.
- 본문 `grep 키워드:` 토큰은 **검색 커버리지 정본 보유처**다 — 카테고리 README 는 (curate 환원 후) 검색 정본이 아니라 레시피 포인터 `grep -rlF` 만 가지므로, 다음 에이전트가 `grep -rlF '<키워드>' docs/compound/{카테고리}/` 로 사례를 찾을 수 있도록 본문 키워드를 구체적으로 작성한다(라우팅은 유형 파일 첫 줄 `> load-when:` 술어가 담당하고, `grep 키워드:` 는 검색용이다 — 둘은 다른 메커니즘).
