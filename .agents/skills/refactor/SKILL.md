# 리팩토링

## 목적

기존 동작을 보존하면서 코드 구조를 정리한다.

이 스킬은 다음 상황에만 사용한다.

- 중복 코드 제거
- 함수 또는 파일 분리
- 책임 경계 정리
- 이름 개선
- 조건문 단순화
- 테스트 보강
- 사용하지 않는 코드 제거

## 실행 순서

1. 입력 검증
2. 현재 구조 파악
3. 변경 범위 축소
4. 리팩토링 설계
5. 최소 변경 적용
6. 검증
7. 결과 보고

## 판단 기준

- 동작 변경 가능성이 있으면 수정하지 않는다.
- 기능 추가가 섞이면 중단한다.
- 추측으로 구조를 바꾸지 않는다.
- 한 번에 넓은 범위를 건드리지 않는다.
- 필수 파라미터 검증과 엣지 케이스 처리를 먼저 정리한다.

## 변경 원칙

- KISS를 우선한다.
- DRY를 우선한다.
- YAGNI를 우선한다.
- SRP를 우선한다.
- 기존 테스트를 깨지 않도록 변경한다.
- 테스트가 없으면 회귀 위험이 큰 경로부터 보강한다.

## 테스트 원칙

- 테스트 코드는 Given-When-Then 패턴으로 작성한다.
- 정상, 경계, 실패 케이스를 함께 확인한다.
- 리팩토링 후에는 기존 동작 보존을 검증한다.

## 협업 기준

- `orchestrate`에서 전달된 구조 개선 요청을 처리한다.
- 코드 품질 검증은 `qa`가 담당한다.
- 요구사항 충족 검증은 `evaluate`가 담당한다.
- 구조 변경이 기능 변경으로 보이면 즉시 중단하고 상위 에이전트에 재배치를 요청한다.

## 보고 기준

- 변경 파일 목록을 명시한다.
- 변경 이유를 한 줄로 요약한다.
- 기존 동작 보존 여부를 명시한다.
- 실행한 테스트와 결과를 적는다.
- 확인이 필요한 항목이 있으면 분리해서 적는다.
