## 재발 방지 학습 (작업 전 필독)

이 영역의 과거 버그 진단 사례는 `docs/compound/bugfix/README.md` 인덱스에 유형별로 정리돼 있다(해당 경로가 없으면 아직 누적된 학습이 없는 것이므로 건너뛰고 진행한다). 경로가 있으면 작업 시작 전 그 README를 읽고, 각 유형 파일 **첫 줄 load-when 술어**를 확인하여 현재 증상이 해당하는 사례 파일만 로드하라(전부 읽지 말 것). 사례를 이 SKILL.md에 누적하지 않는 이유는 스킬 로딩 컨텍스트를 상수 크기로 유지하기 위함이다 — compound-learner가 `docs/compound/`에만 기록한다(load-when 술어·인덱스 형식 규범은 `compound` 스킬의 `### Phase 2: docs/compound 작성` 참조).

## 실행 주체

오케스트레이터가 구현 컨텍스트를 유지한 채 이 스킬을 직접 사용한다.

## 진단 절차

### Step 1. 증상 명확화

사용자 보고에서 다음을 파악한다:
- **무엇이** 잘못됐는가 (증상)
- **어디에서** 발생하는가 (URL, 컴포넌트, 조건)
- **언제부터** 발생했는가 (특정 변경 후인지)

정보가 부족하면 수정 전에 사용자에게 질문한다.

### Step 2. 원인 탐색 (계층 순서)

아래 순서대로 좁혀간다. 해당 계층에서 원인이 발견되면 이후 계층은 확인 불필요.

```
1. 마크업/렌더링 — 요소가 조건부로 숨겨지는지 확인
   └─ 비렌더(null 반환), 표시 토글, 가시성/투명도로 인한 숨김
      예(프로젝트별): null 반환, display:none, visibility:hidden, opacity:0

2. 스타일 — 스타일이 적용되지 않는지 확인
   └─ 쌓임 순서 충돌, 배치 문제, 컨테이너 클리핑,
      기본 스타일 덮어쓰기 누락
      예(프로젝트별): z-index 충돌, position 문제, overflow:hidden, <dialog> 기본 border

3. 클라이언트 로직/이벤트 — 상태 업데이트가 발생하는지 확인
   └─ 상태 갱신 미호출, 이벤트 전파 제어 과잉, 효과/구독 의존성 오류
      예(프로젝트별): setState 미호출, stopPropagation 과잉, useEffect 의존성

4. 서비스/API 응답 — 데이터가 올바르게 오는지 확인
   └─ 빈 배열, null, 잘못된 필드명

5. 데이터 접근 레이어 — 데이터가 올바르게 조회되는지 확인
   └─ 조회 조건, 결합(JOIN) 오류, 파라미터 바인딩
```

### Step 3. 수정 범위 보고

원인이 확정되면 수정 전에 보고한다:
```
원인: {한 줄 요약}
수정 파일: {경로:라인}
수정 내용: {변경 사항 요약}
```

파일 수 자체가 아니라 변경 위험으로 확인 게이트를 정한다. 공개 API·DB schema·인증/권한·데이터 손실 가능성에 닿거나 사용자 요청 범위를 확장해야 하면 수정 전에 `needs-input`으로 사용자 승인을 요청한다.

### Step 4. 최소 수정 적용

- 버그 수정에 필요한 코드만 변경한다
- 인접 코드 리팩터링·정리 금지
- 프로젝트 AGENTS.md가 정한 컨벤션 준수
- 기존 디자인 토큰 유지

### Step 4.5. 자가 검증 (회귀 확인)

수정 직후, 수정 결과의 회귀 여부를 저비용으로 자가 확인하고 그 결과를 Step 5 `수정 완료` 본문에 1줄 증거로 포함한다. 전체 테스트 스위트가 아니라 수정 파일에 **대응하는** 테스트만 다룬다.

- **대응 테스트가 있으면**: 그 테스트 파일만 실행하고, 러너 요약 라인 원문(예: `Tests: 12 passed`)을 증거로 인용한다.
- **대응 테스트가 없으면**: 조건부 렌더·분기가 보존됐는지를 변경 diff로 자가 확인하고, 보존 사실을 1줄로 명시한다(예: `diff 확인 — 조건부 렌더 분기 변경 없음`).

### Step 5. 수정 결과 보고

```
수정 완료:
- 파일:라인 — 변경 요약

원인: {한 줄}
재현 조건: {수정 전 증상을 재현한 입력·상태}
재발 조건: {같은 결함이 다시 나타나는 조건 또는 없음}
검증 증거: {실행 명령·종료 코드·러너 요약 또는 diff 확인 결과}
미검증 항목: {없음 또는 검증하지 못한 항목과 이유}

구조적 문제 (수정 제외):
- {문제 설명} → orchestrate로 처리 권장

공통 반환: `orchestrate/references/agent-contract.md`
에이전트: orchestrator
재현 확인: <confirmed | not-reproduced | environment-blocked>
원인: <원인 또는 미확정>
재발 조건: <조건 또는 미확정>
evaluator 필요 여부: <required | skip-visual-only | 미확정>
```

결과는 다음처럼 판정한다. 수정과 대응 검증이 끝났으면 `completed`, 재현·요구사항 확인이 필요하면 `needs-input`, 환경·권한 때문에 진행할 수 없으면 `blocked`, 수정 또는 검증이 실행됐으나 실패한 채 종료하면 `failed`다. `needs-input`, `blocked`, `failed`를 완료로 표현하지 않는다.

## 스코프 제한

| 처리 | 미처리 (보고만) |
|------|----------------|
| 단일 컴포넌트 렌더링 오류 | 전체 아키텍처 변경이 필요한 구조적 문제 |
| CSS 속성 누락·오타 | 디자인 시스템 개편 |
| 이벤트 핸들러 오작동 | 새 기능 추가 |
| API/서비스 응답 오류 | 데이터 스키마 변경 |
| 데이터 조회 쿼리 오류 | 리팩터링 |

## 주의

- **재현 불가 버그**: 가설을 세우고 수정 전 사용자 확인을 요청한다.
- **마일스톤 중 발생 버그**: orchestrate가 같은 구현 컨텍스트에서 이 스킬을 사용하고 공통 계약으로 결과를 정리한다.
