## 재발 방지 학습 (작업 전 필독)

이 영역의 과거 gap 사례는 `docs/compound/evaluation/README.md` 인덱스에 유형별로 정리돼 있다(해당 경로가 없으면 아직 누적된 학습이 없는 것이므로 건너뛰고 진행한다). 경로가 있으면 작업 시작 전 그 README를 읽고, 각 유형 파일 **첫 줄 load-when 술어**를 확인하여 현재 검증이 해당하는 사례 파일만 로드하라(전부 읽지 말 것). 사례를 이 SKILL.md에 누적하지 않는 이유는 스킬 로딩 컨텍스트를 상수 크기로 유지하기 위함이다 — compound-learner가 `docs/compound/`에만 기록한다(load-when 술어·인덱스 형식 규범은 `compound` 스킬의 `### Phase 2: docs/compound 작성` 참조).

## 검증 원칙

코드 품질(qa-guard 담당)이 아니라 요구사항 대비 구현의 gap을 찾는 것이 목적이다. "코드가 존재한다"는 것만으로 통과시키지 않는다.

## 책임 경계

evaluator는 요구사항 충족 여부와 사용자 플로우 연결만 판정한다.

| 담당 | evaluator 처리 |
|---|---|
| 코드 품질, SQL injection, 테스트 실패 | qa-guard 보고서를 참조하고 중복 판정하지 않는다 |
| 서버 강제 검증, IDOR, 세션·토큰·쿠키·webhook, 민감정보 공격면 | security-auditor 보고서를 참조하고 중복 판정하지 않는다 |
| 색상, 간격, 반응형, hover/focus | design-reviewer 보고서를 참조하고 중복 판정하지 않는다 |
| 스펙 누락, 제거 대상 초과 구현, 링크 단절, 호출되지 않는 API | evaluator가 직접 판정한다 |

원본 스펙 또는 티켓에 명시된 관리자·소유자·조직·테넌트 권한 조건이 구현에 존재하고 플로우에 연결됐는지는 evaluator가 직접 판정한다. 스펙에 명시되지 않은 위협 모델은 security-auditor 담당이다.

검증 중 기술 품질 문제가 발견되면 요구사항 평가와 분리하여 "qa-guard 전달 필요"로 기록한다. 보안 위험이 의심되면 "security-auditor 전달 필요"로 기록한다.

## 0. 티켓 기반 체크리스트 파생 (모든 마일스톤 필수)

3자 대조에 앞서 가장 먼저 실행한다. 사전에 알려진 gap 패턴에 의존하는 대신, **지금 검증하는 작업의 티켓 자체에서 체크리스트를 파생**한다. 어떤 작업이든 이 단계를 거치면 그 작업에 맞는 검증 기준이 자동으로 생성된다.

**읽을 파일**: `docs/02-roadmap/tickets/m{N}-{slug}.md`. 티켓 문서가 없으면 "근거 부재(티켓 미존재)"로 명시하고, 마일스톤 스펙·구현 코드만으로 체크리스트를 파생해 진행한다.

### 단계 A — 산출물 목록 추출

각 티켓의 **산출물** 섹션에 열거된 파일 경로·함수명·API 엔드포인트를 전부 추출하여 아래 표로 변환한다.

| 산출물 항목 | 종류 | 존재 확인 | 연결 확인 |
|---|---|---|---|
| `src/foo/bar.js` | 파일 | 파일 실존 | import하는 곳이 있는가 |
| `GET /api/items` | 엔드포인트 | 핸들러 파일 존재 | 호출하는 클라이언트 코드 |
| `createFoo(no, payload)` | 함수 | 정의 위치 | 호출 지점 |

**"존재만 확인" = `△ 부분`**. 존재 + 실제 연결이 모두 확인된 경우에만 `✓ 충족`.

### 단계 B — 검증 기준 코드 추적

각 티켓의 **검증 기준** 섹션의 Given-When-Then 구문을 하나씩 코드 경로로 추적한다.

| 구문 | 추적할 내용 |
|---|---|
| **Given** | 초기 조건(세션 유무, DB 상태, 파라미터)을 코드가 어떻게 전제하는가 |
| **When** | 트리거(API 호출, 버튼 클릭, 상태 변경)가 어느 파일·함수에서 발생하는가 |
| **Then** | 기대 결과(렌더링 변화, DB 기록, 리다이렉트)가 실제로 코드에 연결됐는가 |

추적 도중 한 단계라도 코드 경로가 끊기면 `✗ 누락` 또는 `△ 부분`으로 표시한다.

`? 확인 필요`는 아래 조건을 **모두 만족**하는 경우에만 사용한다:
1. 코드 정적 분석으로는 동작 여부를 판단할 수 없고
2. 브라우저 스모크 체크(단계 C)를 수행했으나 해당 플로우를 재현할 수 없거나
3. 외부 서비스 연동(OAuth, 결제, 알림 발송 등) 타이밍 의존 동작인 경우

**`? 확인 필요` 남용 금지**: 렌더링 결과, 상태 전환, 데이터 표시 — 브라우저에서 확인 가능한 항목은 단계 C를 먼저 수행한 뒤 `✓` 또는 `✗`로 판정한다. "브라우저 확인이 번거롭다"는 이유로 `?`를 사용하지 않는다.

> 이 단계의 결과물이 이후 3자 대조(축 1~4)의 기준 항목 목록으로 사용된다.

### 단계 C — 브라우저 스모크 체크

단계 A·B에서 파악한 핵심 플로우를 실제 브라우저(브라우저 자동화 도구, 있을 경우)로 확인한다. 정적 분석으로 `✓`라 판단한 항목도 반드시 여기서 런타임 동작을 검증한다. 브라우저 자동화 도구가 없으면 이 단계를 건너뛰고 단계 A·B의 정적 분석 결과로 판정하되, 런타임 확인이 필요한 항목은 `? 확인 필요`로 표기한다.

**전제 조건**: 프로젝트 dev 서버가 실행 중인지 확인한다. 실행 중이 아니면 백그라운드로 기동한 뒤 로컬 주소가 응답할 때까지 대기한다.

**수행 절차**:

1. 브라우저 자동화 도구로 새 탭을 연다
2. 아래 체크 항목을 순서대로 수행하며 결과를 기록한다:

| 체크 항목 | 수행 방법 | 판정 기준 |
|---|---|---|
| 페이지 로드 에러 없음 | 각 구현 페이지 URL 접근 후 콘솔 메시지로 에러 확인 | 500 에러·uncaught exception 없으면 통과 |
| 핵심 UI 요소 렌더링 | 요소 탐색 도구로 핵심 요소(테이블, 카드, 폼) 존재 확인 | 요소가 DOM에 존재하면 통과 |
| 조건부 분기 UI | 분기 조건을 변경(토글·탭 클릭)하며 대응 UI 변화 확인 | 양방향 분기 모두 예상 UI가 렌더링되면 통과 |
| 데이터 표시 | 실제 데이터가 목록·상세에 표시되는지 확인 | 더미 텍스트 없이 DB 데이터가 렌더링되면 통과 |
| 핵심 액션 완료 | 폼 제출·버튼 클릭 후 결과 확인 | 성공 응답·리다이렉트·DB 반영이면 통과 |
| 테이블 행 액션 전수 | 검증 범위에 동적 테이블이 포함되면 축 5 인벤토리의 각 행 액션(수정/삭제/상세/복사/승인 등)을 1행에서 1회씩 클릭하여 네트워크·페이지 전환 확인 | 모든 액션이 4xx/5xx 없이 의도된 라우트·API로 연결되면 통과 |

3. 체크 중 실패 항목 발견 시:
   - 즉시 `✗ 누락` 또는 `△ 부분`으로 등급을 조정한다
   - 정적 분석에서 `✓`라 판단했던 근거를 재검토하여 오판 원인을 보고서에 기록한다

**수행 기록 동봉 (의무)**: 수행한 체크 항목마다 방문 URL·수행 액션·관찰 결과를 1줄로 보고서의 `## 스모크 체크 수행 기록` 섹션에 기재한다. 이 단계를 건너뛴 경우에도 섹션을 비우지 않고 생략 사유 1줄(예: "생략 — 브라우저 자동화 도구 부재")을 기재한다(검증자도 LLM이다 — 스모크를 수행했다는 주장과 실제 수행을 산출물로 구분하는 증거 계약이며, 섹션이 비면 오케스트레이터가 보고서 재전송을 요구한다).

**스모크 체크 범위**: 구현된 모든 페이지를 전수 검사하지 않는다. 티켓의 **검증 기준** Given-When-Then 항목마다 최소 1회 브라우저 동작 확인을 수행한다. 인증이 필요한 페이지는 로그인 후 세션 쿠키를 유지한 상태로 진행한다.

## 3자 대조 방법

### 축 1: 마일스톤 스펙

`docs/02-roadmap/01-milestones.md`와 `docs/02-roadmap/02-work-tickets.md`에서 해당 마일스톤의 **산출물 목록**을 추출한다.

추출 형식:
```
[ ] 홈 / (페이지 구현)
[ ] 목록 / (검색 + 카테고리 + 지역 필터)
[ ] 상세 / (조회수 기록 포함)
...
```

### 축 2: 원본 스펙 / 레거시 소스 (구현 에이전트 산출물, 있을 경우)

`_workspace/01_legacy_*.md`의 "유지/제거 기능 분류표"에서 아래를 확인한다. 레거시 분석 산출물이 없으면(레거시 소스가 애초에 없는 신규 프로젝트 등) 이 축을 생략하고 마일스톤 스펙↔구현 코드 2자 대조로 진행한다.

- **유지 결정 기능**: 구현 코드에 존재하는지 확인
- **제거 결정 기능**: 구현 코드에 없는지 확인 (존재하면 ⚠ 초과로 처리, 정상적으로 없으면 ◐ 의도적 제외로 "의도적 제외 목록"에 기재)

### 축 3: 구현 코드

파일 목록을 읽어 실제 구현 항목을 파악한다.

- 파일 존재만으로 "구현됨" 판정하지 않는다
- 라우트 연결, 조건 분기, 데이터 흐름이 끊기지 않는지 확인한다
- 사용자 플로우 시나리오(`docs/04-legacy-public/flows/`, 있을 경우)를 기준으로 처음~끝을 추적한다

### 축 4: 네비게이션 링크 인벤토리

구현된 UI의 모든 네비게이션 링크가 실제 페이지로 연결되는지 확인한다.

**대상 컴포넌트**: 사이드바·헤더·푸터·하단 네비 등 내부 링크(`href` 속성)를 포함한 모든 레이아웃/네비게이션 컴포넌트.

**절차**:
1. 대상 컴포넌트에서 내부 링크(`href` 값이 `/`로 시작하는 것) 전체 추출
2. 각 href에 대해 대응하는 라우트/페이지 파일 경로를 프로젝트의 라우팅 규칙에 따라 계산
3. 파일 존재 여부 확인 후 아래 표로 보고:

```
| href | 라우트/페이지 경로 | 상태 | 예정 마일스톤 |
|------|------------|------|------------|
| /admin/campaigns | <대응 페이지 경로> | ✓ 구현 | - |
| /admin/recruitment | <대응 페이지 경로> | ✗ 미구현 | M{N} |
```

**마일스톤 단계별 심각도**:

| 마일스톤 범위 | 미구현 링크 처리 | 이유 |
|---|---|---|
| 초기/목업 단계 | `? 확인 필요` — non-blocking | 사이드바·탭 껍데기는 의도적 미구현 허용 |
| 실데이터 단계 이후 | `✗ 누락` — blocking | 링크가 있으면 페이지도 있어야 함 |

외부 링크(`http://`, `#`)와 API 경로(`/api/`)는 검사 대상에서 제외한다.

### 축 5: 테이블 액션 인벤토리

구현된 데이터 테이블(목록·관리 화면의 동적 표) 안의 모든 행/열 액션이 원본 스펙(레거시 소스, 있을 경우)의 컬럼 render와 1:1 매칭되는지 대조한다. 축 4는 레이아웃 네비게이션 링크만 보지만, 사용자 작업의 절반 이상은 테이블 행 안의 액션(수정/삭제/상세/복사/승인/엑셀 등)에서 발생한다.

**전제 조건**: 구현 에이전트의 레거시 분석 산출물(`_workspace/01_legacy_*.md`)에 **Datatable 컬럼/액션 인벤토리** 섹션이 있어야 한다. 없으면 evaluator가 검증 불가로 표시하고 레거시 분석 재호출을 요청한다. 레거시 소스 자체가 없는 신규 프로젝트면 마일스톤 스펙의 테이블 요구사항을 기준 목록으로 삼아 대조한다.

**대상**: 구현 코드에서 행이 동적으로 채워지는 모든 표 — 공용 데이터 테이블 컴포넌트, 페이지 컴포넌트 안에서 반복문으로 행을 만드는 표, UI 라이브러리 테이블 사용처 등.

**절차**:
1. 레거시 분석 산출물(또는 마일스톤 스펙)의 컬럼 인벤토리·액션 표를 기준 목록으로 가져온다.
2. 구현 코드에서 대응 테이블의 컬럼 정의와 행 렌더를 추출한다.
3. 컬럼·액션을 1:1로 매핑하여 아래 표로 보고한다.

```
| 원본 컬럼/액션 | 구현 위치 | 트리거 | 타깃 (라우트/API) | 상태 |
|--------------|-----------|--------|-------------------|------|
| 수정 | StoresTable:142 (상세 편집 링크) | href | /admin/stores/[no]/edit | ✓ 충족 |
| 삭제 | StoresTable:155 onClick={() => deleteStore(row.no)} | onClick | DELETE /api/admin/stores/[no] | △ 부분 (확인 다이얼로그 누락) |
| 복사 | - | - | - | ◐ 의도적 제외 (회의록 근거) |
| 가게이름 셀 → 상세 이동 | StoresTable:120 plain text | - | - | ✗ 누락 (원본에서 클릭 시 상세로 이동) |
```

**판정 규칙**:
- 액션이 존재하고 타깃 라우트/API가 응답하면 `✓ 충족`
- 액션은 있으나 호출 API가 미구현(404)·세션 가드 누락·확인 다이얼로그 같은 부속 동작 누락이면 `△ 부분`
- 기준 인벤토리에 있으나 구현 코드에 없으면 `✗ 누락`
- 기준 인벤토리에 있으나 제거 결정된 경우 `◐ 의도적 제외` + 승인 출처 명시
- 권한 조건(관리자 등급 등)이 원본 스펙에 있는데 구현에 빠졌으면 `△ 부분` + 보안 영향 기록

**검증할 핵심 질문**:
- 행 액션 버튼의 `onClick`/`href`가 실제 라우트 또는 API로 연결되는가? (dead handler 금지)
- 호출 API가 존재하고 같은 페이로드 스키마를 받는가? (이름은 같지만 다른 컬럼을 보내는 경우 차단)
- 권한 분기(role/owner check)가 원본 스펙과 동등하게 적용되는가?
- 셀 값 자체가 클릭 가능했던 경우(가게이름 → 상세 이동) 구현에서도 클릭 가능한가?
- 조건부 액션(`status === 'pending'`일 때만 승인 버튼)이 동일 조건으로 노출되는가?

## 평가 기준

| 등급 | 의미 | 처리 |
|---|---|---|
| ✓ 충족 | 스펙 요구사항 완전 구현 + 플로우 연결 확인 | 통과 |
| △ 부분 | 구현되었으나 세부 항목 누락 | 보완 요청 |
| ✗ 누락 | 스펙 요구사항 미구현 | 구현 요청 |
| ⚠ 초과 | 제거 결정 기능이 구현됨 | 제거 요청 |
| ◐ 의도적 제외 | 원본에 존재하나 승인된 결정으로 구현에서 제외됨 | 통과 — 단 "의도적 제외 목록"에 승인 출처와 함께 필수 명시 |
| ? 확인 필요 | 코드만으로 동작 여부 판단 불가 | 사용자 검토 요청 |

**승인된 제외는 통과시키되 침묵시키지 않는다.** `✓ 충족`(원본 충실 구현)과 `◐ 의도적 제외`(원본 대비 차이)를 보고서에서 같은 줄로 합치지 않는다. 오케스트레이터가 "이건 gap 아니다"라고 전달해도 그 항목을 보고서에서 빼는 것이 아니라 "의도적 제외 목록"에 승인 출처와 함께 적는다. 이유: 승인은 통과를 뜻하지 비가시를 뜻하지 않는다. 작업 시작 시점의 추상적 승인이 완성물의 구체적 차이(예: 테이블 열 수 감소)를 가릴 수 있으므로, 최종 감사 산출물에 차이를 영구히 남겨 사후 검토를 가능하게 한다.

## 자주 발견되는 gap 패턴

- 페이지는 있으나 데이터 연결 없음 (더미 데이터 상태로 방치)
- 원본 스펙에서 유지하기로 한 부수 기록(조회수 카운터 등) 누락
- 스펙에 명시된 보호 로직 미구현 (코드 존재해도 권한 조건 미연결)
- 제거 결정했으나 레거시 코드 직역으로 불필요 기능 구현
- 사용자 플로우 중간 단계 누락 (신청 → 가선정 사이 상태 처리 없음)
- 탭 레이아웃만 있고 각 탭 내용 없음
## 산출물 형식

`_workspace/eval_{milestone}_{datetime}.md` 에 아래 구조로 작성한다. 오케스트레이터 봉투가 보고서 파일명을 별도 지정하면(예: 버그 경로 `eval_bugfix_{증상-슬러그}_{datetime}.md`) 기본 명명 대신 그 지정을 따른다(마일스톤 게이트의 파일 개수 기계 카운트 오염 방지).

머리말 `반복 횟수` 기입 규칙(외부 기억 — compaction 후 보완 루프 회차 복원용): 같은 마일스톤의 직전 `_workspace/eval_*` 보고서가 있으면 그 머리말의 N에 +1, 없으면 `1회차`로 기재한다. 버그 경로 보고서(`eval_bugfix_*`)는 이 마일스톤 체인의 대상이 아니다 — 대조·N 계산에서 제외한다. `직전 미해결` 에는 그 직전 보고서의 ✗/△/⚠ 항목(누락·부분·초과 — 보완 루프의 미해결 범위 전체)을 티켓 산출물 항목명 또는 Given-When-Then 구문 키(라인 번호 제외)로 옮겨 적는다 — 직전 보고서가 없으면 `없음`.

```markdown
# 요구사항 검증 보고서: {milestone}

검증 일시: {datetime}
검증 범위: {마일스톤명 + 주요 산출물}
참조 문서:
  - docs/02-roadmap/01-milestones.md
  - _workspace/01_legacy_*.md (사용한 파일 목록)

 승인된 제외라도 ✓ 충족 행에 섞지 않는다 — 원본 대비 차이는 이 목록에서 항상 보이게 둔다.)
```

## 주의사항

- qa-guard 보고서를 참조하되 코드 품질 항목(테스트, SQL injection 등)은 중복 검증하지 않는다.
- security-auditor 보고서를 참조하되 스펙 외 보안 위험 판정은 중복 검증하지 않는다.
- 스펙 문서와 구현 코드 간 불일치가 전체 항목의 30% 이상이면 오케스트레이터에 에스컬레이션한다.
- 마일스톤별 "결정 필요" 항목(프로젝트 요구사항·결정 문서, 있을 경우)이 미결인 채 구현된 경우, 해당 구현의 근거를 확인 필요로 표시한다.
