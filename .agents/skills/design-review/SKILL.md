## 실행 에이전트

design-reviewer

## 핵심 원칙

**체크리스트 기반 합격**: 프로젝트 디자인 레퍼런스가 존재하면 "대충 맞는 것 같다"는 판단 금지. 아래 Step 2~5의 모든 체크리스트 항목이 통과되어야만 `approved`를 반환한다.

**시각 레퍼런스 부재 시 폴백**: 프로젝트에 디자인 레퍼런스(디자인 토큰 카탈로그·시각 스펙)가 존재하지 않으면, 레퍼런스 대조 항목(C-01~C-08)을 전부 N/A로 처리하고 레퍼런스와 무관한 필수 항목인 디자인 토큰 사용·접근성 기초(C-09~C-11)만 검증한다. 레퍼런스 부재 자체는 결함으로 보고하지 않는다.

**검증 수단 독립**: 특정 브라우저·GUI 도구를 합격 조건으로 고정하지 않는다.
오케스트레이터 봉투의 UI 검증 모드와
[UI 검증 수단 폴백](../orchestrate/references/ui-verification-fallback.md)에 따라
브라우저, 기존 E2E, production 컴포넌트 런타임 테스트, HTTP·정적 분석,
사용자 관찰을 요구사항별로 조합한다. 도구 이름이 아니라 증거 커버리지로
판정한다.

**루프 한계**: 직접 수정은 각 변경 뒤 관련 체크를 즉시 재검증한다. 새 증거 없이 같은 FAIL이 반복되거나 수정이 기계적 보정 범위를 벗어나면 내부 반복을 중단하고 오케스트레이터에 반환한다. 외부 보완의 stuck·절대 상한은 `orchestrate/references/error-handling.md`가 소유한다.

**검수-수정 분리**: design-reviewer의 직접 수정은 의미가 바뀌지 않는 기계적 보정으로 제한한다. 레이아웃 구조, 컴포넌트 분해, 조건부 렌더링, 데이터 표시 방식은 구현 에이전트에 위임한다.

**직접 수정 한계**:
- 파일 수나 수정 위치 수만으로 직접 수정 여부를 결정하지 않는다.
- 변경 의미가 불변이고 diff 범위가 명확하며 같은 호출에서 완전 재검증할 수 있을 때만 직접 수정한다.
- 수정 후에는 변경 파일만 재검증하지 않고 원래 검증 대상 전체를 다시 확인한다.

## Step 0. 가이드 섹션 파싱

프로젝트 디자인 레퍼런스(있을 경우)를 Read로 로드하여 검증 대상 컴포넌트의 섹션을 찾는다.
레퍼런스 자체가 없으면 이 Step을 건너뛰고 C-01~C-08을 전부 N/A로 처리한 뒤 Step 3(토큰)·Step 5(접근성)만 수행한다.

**섹션 매핑 방법:**
- 컴포넌트명(영문/한국어)으로 `<h2>`, `<h3>` 등 섹션 헤더 텍스트를 검색한다
- 주요 매핑(예): `Badge` → "뱃지", `Button` → "버튼", `Header` → "헤더"
- 해당 섹션 블록 전체를 추출한다

**추출 항목 (가이드 스펙 객체):**
```
- css_tokens: 가이드 예시에 사용된 디자인 토큰(CSS 변수) 목록
- hover_rules: :hover 시 변경되는 property/value
- focus_rules: :focus/:focus-visible 시 변경되는 property/value
- modifier_classes: 상태 클래스 목록 (e.g., tabActive)
- aria_attributes: 예시 HTML에 포함된 접근성 속성
```

컴포넌트가 가이드에 없으면 → "가이드 미등록" 표시 후 Step 2 체크리스트에서 C-01~C-08을 N/A로 처리한다.

## Step 3. CSS 토큰 미사용 탐지 (C-01~C-05, C-09 판정)

검증 대상 스타일 파일과 프로젝트 전체 스타일 파일을 **Grep 전수 스캔**한다.

```
탐지 패턴 (경고 대상):
- #[0-9A-Fa-f]{3,6}         (하드코딩 hex)
- rgb\(                      (하드코딩 rgb)
- background:\s*#fff         (흰 배경 — 다크모드 핵심 버그)
- background:\s*white
- color:\s*#[0-9A-Fa-f]     (하드코딩 텍스트 색상)
- border.*:\s*#              (하드코딩 테두리 색상)

예외 (수정 불필요):
- calc() 내부 단위
- transform/translate 등 동적 값
- 의도된 고정값으로 주석 처리된 경우: /* design-reviewer: intentional */
- 소셜 미디어 플랫폼 공식 브랜드 컬러 (단, 아래 검증 필수)

**브랜드 컬러 예외 검증 규칙 (Instagram 오공식색 사례)**:
hex가 소셜 플랫폼 브랜드 컬러 목적으로 사용된 경우, 임의 색상이 아닌 공식 브랜드 리소스 기준의 정확한 값인지 반드시 확인한다.
공식 브랜드 컬러 기준표(예시 — 프로젝트가 사용하는 플랫폼에 맞춰 보강):
- Instagram gradient: `linear-gradient(45deg, #feda75, #fa7e1e, #d62976, #962fbf, #4f5bd5)` (5색, 45도) — 임의 보라(`#667eea`, `#764ba2` 등) 사용 금지
- Naver: `#03C75A` (초록)
- Kakao: `#FEE500` (노랑), 텍스트 `#3C1E1E` (갈색)
- YouTube: `#FF0000` (빨강)
브랜드 컬러는 `/* design-reviewer: intentional */` 주석과 함께 출처 URL 또는 색상명을 명시한다. 출처 미명시 브랜드색은 WARN으로 처리한다.
```

**Step 0 비교**: 탐지된 하드코딩 값이 가이드 스펙의 어느 토큰으로 대체되어야 하는지 특정한다.
(단순 "토큰 미사용"이 아니라 "가이드는 배경 디자인 토큰 사용"처럼 어느 토큰으로 대체해야 하는지 구체적으로 보고한다.)

**흰 배경 탐지가 중요한 이유**: `background: #fff`는 다크모드에서 배경이 뒤집혀 시스템 전체와 어긋난다. 신규 파일뿐 아니라 기존 파일도 포함하여 전수 스캔한다.

**C-05 라운드 토큰 교차 불일치 탐지 (커스텀 버튼 radius 불일치 사례)**: hex 하드코딩이 없어도 잘못된 radius 토큰 선택으로 외형 불일치가 발생할 수 있다. 동일 컨텍스트(같은 테이블 행, 같은 액션 컬럼)에 공용 Button과 커스텀 버튼이 공존하는 경우, 두 컴포넌트의 `border-radius` 토큰이 서로 다른지 추가로 확인한다. 탐지 패턴: 커스텀 버튼 스타일에 작은 radius 토큰(예: xs/sm)이 있고, 같은 컨텍스트의 공용 Button이 더 큰 radius 토큰(예: md)을 사용하면 FAIL. 수정 기준: 같은 컨텍스트의 동급 버튼은 동일 radius 토큰으로 통일한다.

## Step 3.5. 인터랙션 상태 캐스케이드 오버라이드 탐지 (C-06, C-08 판정)

**탐지 목적**: 부모/기본 클래스의 `:hover`/`:focus` 규칙이 modifier 클래스의 색상을 덮어쓰는 캐스케이드 버그를 정적 분석으로 잡는다.

**탐지 절차:**

```
1. 같은 스타일 파일에서
   a. `.classA:hover { color/background: X }` 규칙 수집
   b. `.classB { color/background: Y }` (hover 한정자 없는 modifier 클래스) 규칙 수집

2. 대응 마크업/템플릿 파일에서 두 클래스를 같은 element에 조건부 적용하는 패턴 확인
   예: 활성 상태일 때만 modifier 클래스가 기본 클래스에 함께 부여되는 분기

3. .classA:hover 의 property가 .classB 의 property와 동일하고,
   두 클래스가 같은 element에 공존 가능한 경우 → 잠재적 오버라이드

4. .classB에 대응하는 `.classB:hover { property: Y }` 규칙이 없으면 → FAIL
```

**발생 케이스 (참고):**
```css
.tab:hover { color: var(--color-text); }     /* 기본 클래스 — 디자인 토큰 */
.tabActive { color: #fff; }                  /* modifier — :hover 없음 */
/* 결과: 활성 탭에서 hover 시 글자색이 배경색과 같아져 보이지 않음 */
```
→ FAIL: `.tabActive:hover { color: #fff }` 규칙 추가 필요

**판정**: Step 0의 `modifier_classes` 각 항목에 대해 대응 `:hover` 규칙의 존재 여부를 확인한다.

## Step 3.6. 폼 내 선택 컨트롤 시각 언어 일관성 탐지 (C-06, C-08 판정 연계)

**탐지 목적**: 동일 폼 안에서 라디오 칩(알약 형태)과 체크박스가 서로 다른 시각 언어를 사용하는 불일치를 잡는다.

**탐지 절차:**

```
1. 검증 대상 스타일 파일에서 라디오 칩 셀렉터와 체크박스 레이블 셀렉터 수집
   - 라디오 칩 패턴: .toggleButton, .radioChip, .chipLabel 등
   - 체크박스 레이블 패턴: .checkboxLabel, .checkboxItem 등

2. 두 종류가 같은 파일 내 공존할 때 아래 속성을 비교:
   a. border / border-color 토큰
   b. padding 토큰
   c. border-radius 토큰
   d. transition

3. 비교 결과 토큰 또는 값이 다르면 → FAIL
4. .checkboxRow(또는 flex 컨테이너)에 gap이 없으면 → FAIL
5. :has(input:checked) 또는 .checked 상태 강조가 없으면 → FAIL
6. accent-color 토큰 미지정이면 → WARN
```

**판정**: 위 조건 중 하나라도 해당하면 C-08(modifier 상태) FAIL로 분류한다. 수정 방법: 체크박스 레이블에 라디오 칩과 동일한 보더/패딩/라운드/전환 스타일 적용, `:has(input:checked)` 강조 추가.


## Step 6. 체크리스트 합격 판정

C-01 ~ C-11 전 항목 결과를 집계한다.

```
합격 기준:
- N/A가 아닌 모든 항목이 PASS → approved
- FAIL이 1건 이상이면 → Step 7로 이동
- 런타임 확인이 필수인 항목에 등가 증거가 없으면 → approved 금지

WARN(경고)으로 처리하고 approved 가능한 경우:
- 의도된 예외로 주석 처리된 경우: /* design-reviewer: intentional */
- 가이드에 해당 컴포넌트 섹션 자체가 없는 경우 (N/A)
```

실제 hover/focus, viewport별 overflow, 동적 콘텐츠 겹침처럼 런타임에서만
확인 가능한 항목은 폴백 문서의 증거 등가표로 확인한다. 정적 분석은 결함을
확정할 수 있지만 시각 정상 동작을 단독 증명하지 못한다. completion 모드에서
등가 증거가 끝내 없으면 사용자 시각 확인으로 해소할 수 있을 때 `needs-input`,
필수 환경 자체가 없을 때 `blocked`다. audit 모드에서는 해당 항목을 `미확인`과
검증 부채로 기록하고 `completed + 수정필요`로 보고서를 완성해 후속 진단을
막지 않는다.

로그인, 결제, 데이터 삭제 등 고영향 동작은 UI 도구가 제공되더라도 자동
수행하지 않고 사용자 확인을 유지한다.

## Step 7. 반복 루프 결정

FAIL 항목 유형에 따라 처리 방식을 결정한다.

| FAIL 유형 | 처리 주체 | 재검증 단계 |
|----------|---------|-----------|
| CSS 토큰 치환 (hex → var) | design-reviewer 직접 수정 | Step 3 재실행 |
| :hover 캐스케이드 오버라이드 규칙 추가 | design-reviewer 직접 수정 | Step 3.5 재실행 |
| aria-label / alt 추가 | design-reviewer 직접 수정 | Step 5 재실행 |
| 레이아웃 구조 변경 필요 | 구현 에이전트 위임 | 구현 에이전트 완료 후 Step 4 재실행 |
| 반응형 리팩터링 필요 | 구현 에이전트 위임 | 구현 에이전트 완료 후 Step 4 재실행 |
| 컴포넌트 구조 변경 필요 | 구현 에이전트 위임 | 구현 에이전트 완료 후 Step 2~5 재실행 |
| 데이터 표시 조건 변경 필요 | evaluator 또는 구현 에이전트 위임 | 담당 보완 후 Step 2~5 재실행 |

직접 자동 수정한 파일은 수정 직후 표의 대응 단계만 재실행한 뒤, **변경된 전체 scope에 대해 Step 2~6을 다시 실행**한다. 최종 보고서의 자동 수정 내역에는 파일과 재검증 증거를 기록하고, 후속 qa-guard에 정확한 수정 파일 목록을 전달한다. 재검증 전에는 `approved`를 반환하지 않는다.

`approved` 반환 시까지 반복한다. 동일 FAIL 항목 목록이 **3회 연속** 변화 없으면 stuck 판정 후 에스컬레이션한다 (구현 에이전트 위임을 거치는 이 외부 보완 루프의 3회 무변화·절대 라운드 상한 판정은 오케스트레이터 소유다 — "핵심 원칙 > 루프 한계" 참조. 아래 대조 규칙은 그 판정의 기준 정의다):
- 매 반복마다 FAIL 항목 ID 목록을 직전 회차와 대조한다
- 새로 해결된 FAIL 항목이 1건 이상 있으면 카운터를 리셋하고 루프를 계속한다
- 변화 없이 stuck 기준에 도달하면 에스컬레이션한다. 판정 주체는 오케스트레이터다.
```
⚠ design-reviewer 에스컬레이션
3회 연속 동일 FAIL 항목 미해결 — stuck 상태
미해결 FAIL 항목: <목록>
원인: <각 항목별 원인>
권고: 사용자 직접 검토 필요
```

`approved`는 `결과: completed`에서만 가능하다. completion 모드에서 사용자
판단이 필요하면 `needs-input`, 대체 불가능한 실행 환경 부재는 `blocked`, 자동
수정이나 재검증 자체의 실패는 `failed`로 반환한다. audit 모드에서는 가능한
폴백을 모두 수행하고 미확인 범위를 특정했다면 `completed + 수정필요`를
반환할 수 있으며, 이는 구현 완료 승인이 아니다.

## 검증 제외 항목

아래는 qa-guard 담당이므로 중복 검증하지 않는다:
- 프로젝트 AGENTS.md가 정한 라이브러리/의존성 화이트리스트 준수
- 금지된 언어·파일 형식 사용 여부
- SQL injection / 보안
- 컴포넌트 위치·파일 배치 규칙 위반
