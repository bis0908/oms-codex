# 에이전트 공통 계약

## 작업 봉투

```text
request_id: <호출별 고유 ID>
마일스톤: M{N | N/A}
목표: <사용자 가치 1문장>
범위: <파일·기능 목록>
제외: <목록 | 없음>
참조: <정확한 문서·보고서 경로 | 없음>
완료 기준: <검증 가능한 조건>
task_id: <목록 | 없음>
checklist_id: <목록 | 없음>
위험 클래스: <none | ui | data | auth | payment | migration | destructive>
금지 사항: <목록 | 없음>
UI 검증 모드: <completion | audit | 해당 없음>
선호 UI 검증 수단: <사용 가능한 도구·runner | 자동 선택 | 해당 없음>
대체 검증 허용: <예 | 아니오 | 해당 없음>
```

`목표`, `범위`, `완료 기준`은 필수다. 선택 필드는 비워 두지 않고 `없음`을
사용한다. UI를 확인하는 모든 agent에는 UI 검증 모드, 선호 수단, 대체 검증
허용 여부와 URL·viewport·재현 시나리오를 포함한다. 특정 도구를 고정하지
않으며 실제 수단은 [ui-verification-fallback.md](ui-verification-fallback.md)에
따라 선택한다. 검증 agent에는 앞선 보고서의 정확한 경로와 같은 `request_id`를
전달한다.

## 공통 반환

모든 구현·검증·학습·추적 agent는 다음 키를 유지한다.

```text
결과: <completed | needs-input | blocked | failed>

완료 항목:
- <파일·검증·상태 전이 | 없음>

완료 task_id:
- <task_id | 없음>

미완료 항목:
- <항목 | 없음>

검증:
- <명령, exit code, 결과 분류 | 없음>

미검증 항목:
- <항목과 사유 | 없음>

확인 필요:
- <질문 | 없음>

다음 단계:
- <오케스트레이터가 수행할 작업>

마일스톤: M{N | N/A}
단계: <phase>
에이전트: <agent-name>
request_id: <입력과 같은 ID>
```

역할별 필드는 이 블록 뒤에 추가한다. 공통 키를 바꾸거나 생략하지 않는다.

## 상태 의미

| 결과 | 의미 | 다음 처리 |
|---|---|---|
| `completed` | 완료 기준과 필수 검증 충족 | 다음 게이트 진행 |
| `needs-input` | 사용자·정본 결정 필요, 파일 변경 없음 또는 안전한 부분 결과만 존재 | 질문 후 재호출 |
| `blocked` | 환경·필수 검증·의존성 때문에 완료 불가 | 파이프라인 중단 |
| `failed` | 실행 자체 실패 또는 계약 위반 | 1회 재시도 후 필수 역할은 중단 |

`미검증 항목`이 있으면 그 항목이 필수인지 역할별 결과 필드에 명시한다.
completion 모드의 필수 미검증을 `completed`로 반환하지 않는다. audit 모드는
가용한 폴백을 모두 수행하고 미검증 범위와 해소 조건까지 분류한 경우 감사
산출물을 `completed`로 반환할 수 있지만, 역할별 결과는 `수정필요`이며 구현
완료 신호로 사용할 수 없다.

## Task 소유권

- task 상태 저장소와 checklist 매핑은 오케스트레이터만 수정한다.
- 하위 agent는 입력으로 받은 task ID와 완료 task ID만 반환한다.
- 오케스트레이터는 파일 실존, diff, 검증 증거를 확인한 뒤 completed로 전이한다.
- milestone-tracker는 명시된 checklist ID만 갱신한다. 파일명으로 항목을 추측하지 않는다.

## 산출물 명명

- 시간대: KST
- datetime: `YYYYMMDD-HHmmss+0900`
- scope: 영문 소문자·숫자·하이픈만 사용
- 같은 이름이 있으면 덮어쓰기가 명시된 형식 재전송 외에는 `-02`, `-03` 순번을 붙인다.
