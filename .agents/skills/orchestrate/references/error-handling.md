# 에러 핸들링 — orchestrate

## 기본 원칙

- 에이전트 실패 시 1회 재시도, 재실패 시 해당 결과 없이 진행하고 보고서에 명시
- 상충 정보 발견 시 삭제하지 않고 출처를 병기

## 보완-재검증 루프 전략

evaluator, qa-guard, design-reviewer가 실패/누락 발견 → 담당 에이전트에 보완 요청 → 재검증 → approved/전 항목 통과 시까지 반복

**학습 트리거**: qa-guard 또는 design-reviewer의 동일 미해결 항목이 2회 연속 반복되면 compound-learner를 호출한다
- qa-guard 반복: `트리거: qa-repeat`, 보고서: 최신 `_workspace/qa_*.md`
- design-reviewer 반복: `트리거: design-repeat`, 보고서: 최신 `_workspace/design_review_*.md`
- 동일 반복 묶음은 1회만 학습한다. 다음 반복에서 미해결 항목 목록이 바뀌면 새 묶음으로 본다.
- compound-learner 실패는 검증 루프를 중단하지 않는다. 실패 사유를 완료 보고에 포함한다.

## Stuck 감지

이 절은 하네스 전체의 **stuck 임계 단일 정본(single source)**이다. 경로마다 임계 수치가 다르며, 다른 이유는 경로의 신호 성질이 다르기 때문이다. 임계 자체보다 **의도 표기**가 핵심이다 — 수치를 베끼지 말고 이 표를 참조한다.

| 경로 | stuck 임계 | 왜 이 수치인가 |
|---|---|---|
| 파이프라인 검증 루프 (qa-guard·evaluator·design-reviewer) | 동일 미해결 3회 연속 + 통산 7회 절대 라운드 상한 | 게이트는 부분 진전이 가능하다 — 일부 항목이 풀리면 카운터를 리셋하므로 3회까지 여유를 둔다. 리셋이 영구 반복되는 진동·매회 신규 gap 출현은 절대 상한이 잡는다(하단 "절대 라운드 상한" 절) |
| Phase 3-2 원자 작업 검증 | 동일 태스크 2회 연속 미완료 | 현재 세션 작업 목록은 완료/미완료 이진 신호라 부분 진전 개념이 없다 — 2회면 진행 불가가 확정된다 |
| 버그 경로 evaluator 재검증 | 동일 미해결 2회 연속 | 단발 경로라 마일스톤 파이프라인과 무관하다 — 빠른 반환이 저렴하므로 2회에서 끊는다 |

아래 절차는 파이프라인 검증 루프(3회 기준)에 적용한다. Phase 3-2·버그 경로는 위 표의 자체 임계를 따르되 동일성 판정 규칙은 공유한다.

동일한 미해결 항목 목록이 3회 연속으로 변화 없이 반복되면 stuck으로 판정한다.

- 매 반복마다 미해결 항목 목록을 **안정 키**로 직전 회차와 대조한다. 안정 키는 보완 수정으로 파일이 바뀌어도 동일 결함을 같은 항목으로 식별하기 위한 것이다.
  - design-reviewer: 체크 항목 식별자(C-ID)
  - qa-guard: 체크 섹션명 + 파일 경로(라인 번호 제외)
  - evaluator: 티켓 산출물 항목명 또는 Given-When-Then 구문
  - **라인 번호는 동일성 판정에 사용하지 않는다.** 보완 수정은 같은 결함의 라인을 매회 이동시키므로, 라인을 키에 넣으면 동일 항목이 "목록 바뀜"으로 오인되어 카운터가 영원히 임계에 도달하지 못한다.
- 새로 해결된 항목이 1건 이상 있으면 카운터를 리셋하고 루프를 계속한다
- 변화 없이 3회 연속이면 에스컬레이션한다:

```
⚠ 에스컬레이션: M{N} {항목명}
원인: {3회 연속 동일 항목 미해결 — stuck 상태}
미해결 항목: <목록>
권고: 사용자 직접 검토 필요
```

에스컬레이션 후 해당 항목은 "보류"로 표시하고 나머지 항목의 진행을 계속한다.

### 절대 라운드 상한 (무변화 감지의 사각 보완)

무변화 3회 감지는 회귀 진동(A→B→A)과 회차마다 새 gap이 출현하는 경우를 잡지 못한다 — 매회 "해결 1건"이 카운터를 리셋하고, 검증자는 매 호출이 신선 컨텍스트의 LLM 판정이라 회차마다 다른 항목을 보고할 수 있다. 그래서 파이프라인 검증 루프에는 무변화 여부와 무관한 절대 상한을 함께 둔다.

- **임계**: 동일 게이트의 검증(최초 검증 + 보완→재검증)이 통산 **7회(튜닝 가능)** 를 초과하게 되면 stuck으로 판정하고 위 에스컬레이션 형식을 사용한다. 원인 줄에는 `절대 라운드 상한 초과 — 진동 또는 회차별 신규 gap` 을 기재한다.
- **카운트 방법**: 회차는 LLM 기억이 아니라 현 `_workspace` 의 해당 게이트 보고서 파일 개수를 Glob으로 세는 **기계 카운트**다(검증 1회차마다 보고서 파일이 1개씩 생성되므로 파일 개수 = 통산 회차. 파일 기반이라 compaction 후에도 보존된다). 보고서 머리말 `반복 횟수` 와 파일 개수가 어긋나면 파일 개수를 우선한다(기계 카운트가 정본).
  - evaluator: `_workspace/eval_{해당 마일스톤}_*.md`
  - qa-guard: `_workspace/qa_{해당 feature}_*.md`
  - design-reviewer: `_workspace/design_review_{해당 scope}_*.md` (보고서 머리말 `반복 횟수: {N}/5` 는 단일 호출 내 내부 수정 루프 카운터이므로 외부 회차로 쓰지 않는다)
- **마일스톤 단위 통산**: `_workspace` 는 새 마일스톤 시작 시 `_workspace_prev/` 로 이동하므로(orchestrate "_workspace 상태 분기") 카운트는 마일스톤 단위로 자연 리셋된다 — 별도 초기화 절차가 필요 없다.
- **네임스페이스 오염 방지**: 버그 경로 evaluator 보고서는 같은 `eval_*` 네임스페이스에 쌓일 수 있으므로 `eval_bugfix_{증상-슬러그}_{datetime}.md` 로 분리 명명한다(orchestrate 버그 경로 3번 참조). 마일스톤 게이트 카운트는 `eval_{해당 마일스톤}_*` 패턴만 센다 — 버그 경로 보고서가 회차를 부풀리지 않게 한다.
- **카운트 비대상**: 게이트 보완 회차가 아닌 보고서는 카운트 패턴에 들어오지 않게 명명·처리한다 — 형식 재전송(직전 보고서와 동일 파일명 덮어쓰기), 증분 재게이팅(`qa_regate_*`/`design_review_regate_*`), 버그 경로(`eval_bugfix_*`), Phase R(`qa_refactor_*`). 각 호출 지점의 작업 봉투가 파일명을 지정한다(orchestrate 해당 절 참조).
- **절차**: 게이트 재검증을 호출하기 전에 개수를 세고, 이미 7개에 도달했으면 재검증을 호출하지 않고 에스컬레이션한다. 호출하는 경우에는 `절대 상한: {게이트} {개수+1}/7` 를 1줄 로깅한다(게이팅 결정 1줄 로깅 패턴 — 사후 감사용).
- **에스컬레이션 후속**: 위 Stuck 감지의 "보류 표시 후 나머지 항목 진행 계속" 규칙을 동일하게 따른다. 보류로 인해 evaluator `approved` 가 성립할 수 없어 Phase 5 진입이 막히면, 이는 컨텍스트 감시의 "2차 보조 — 진행 불가"에 해당하므로 안전 중단하고 사용자 결정을 받는다(진행 계속/세션 중단/대기의 세 해석을 이 우선순위로 단일화한다).
- 이 에스컬레이션은 컨텍스트 중단이 아니라 **루프 종료**이므로, 컨텍스트 감시의 "반복 실패 횟수만으로는 중단하지 않는다" 거짓양성 방지 규칙과 충돌하지 않는다(그 규칙은 정량 임계의 안전 중단 트리거를, 이 절은 보완 루프 자체의 종료를 다룬다).
