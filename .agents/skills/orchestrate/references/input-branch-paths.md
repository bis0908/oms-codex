# 저빈도 입력 분기 처리 경로

> **언제 읽는가**: Phase 0 "입력 유형 분기" 표에서 입력이 **compound 정리/정돈 요청**, **compound 건강 질의**, 또는 **계획 정합성 검증 요청**으로 판별돼 해당 경로로 이탈할 때만 읽는다. 세 경로 모두 마일스톤 파이프라인(Phase 1~5)에 진입하지 않는 저빈도 비파이프라인 진입이므로, 평상시 파이프라인 진행에는 로드하지 않는다. (버그 경로·재개 프롬프트 경로는 본문에 유지 — 빈출이거나 컨텍스트 감시 루프와 결합돼 있음.)

## compound-curate 온디맨드 처리 경로

1. 오케스트레이터 파이프라인(Phase 1~5)을 실행하지 않는다.
2. `compound-learner` 에이전트를 직접 호출한다(인자: 트랙 `compound-curate`, 대상은 요청에 명시된 카테고리/유형 목록이며 없으면 전 카테고리). Phase 5 드레인과 달리 사용자 응답이 필요하므로 백그라운드 실행이 아니라 동기 호출로 진행한다.
3. 반환된 정리 결과(각 `{유형}.md` N→M 활성, 승격 원칙 K줄, 무손실 검증 PASS/FAIL)를 사용자에게 요약 보고하고 종료한다.

> 이유: 누적분 정리는 마일스톤 파이프라인과 무관한 유지보수 작업이다. evaluator `approved` 와 무관하게 발동하며, _superseded 격리/원칙 승격은 compound-learner의 `compound-curate` 트랙만 수행한다(무손실 불변식 — `compound` 스킬 `### 정리 트랙` 절 참조).

## compound 건강 질의 온디맨드 처리 경로

학습 건강(어느 사례가 자주 재발하나·어느 클래스에 자매가 몰렸나)을 **조회·보고**하는 경로다. compound-curate(정리 실행)와 달리 파일을 쓰지 않고 텍스트 표만 출력한다 — 유지되는 `_health.md` 같은 산출물은 그때그때 도구로 새로 파생하면 되는 숫자의 철 지난 사본이 될 뿐이라 만들지 않는다.

1. 오케스트레이터 파이프라인(Phase 1~5)·에이전트를 호출하지 않는다. **파일을 생성하지 않는다**(`_health.md` 금지).
2. 대상 카테고리(요청에 명시 — 없으면 전 카테고리)의 모든 `{유형}.md` 에 대해 도구로 두 지표를 그 자리에서 센다:
   - **재발도**: `grep -c '재발(' {유형}.md`
   - **계보밀도(자매 집중)**: `grep -c '자매(' {유형}.md` (재발도와 대칭 — fixed-string 라인 카운트). 골격이 사례당 방출하는 계보 마커는 `자매(` 뿐이다 — `supersede`·`계보` 는 사례 작성자가 관계를 **서술할 때 쓰는 산문 단어**라 본문에 빈출할 수 있으나(예: `## _superseded` 헤더, `[정본 supersede]` 류 라벨, 관계 설명문) 전부 산문 빈도일 뿐 per-case 구조 마커가 아니다. 산문 단어 빈도를 정렬키에 섞으면 문서의 장황함이 자매 밀도로 둔갑하므로 배제한다.
2.5. **교차-카테고리 자매 재료화 (건강 질의 시 항상 1회 수행).** 위 2번이 카테고리 내부(파일 스코프) 집중만 세는 것의 사각을 메운다 — `자매({기준})` 는 파일 스코프 마커라, 같은 버그 클래스가 두 카테고리(예: frontend + data-layer)에 표면화하면 2번의 per-file 카운트로는 영영 이어지지 않는다. 발동을 "교차를 명시한 질의"로 게이팅하지 않는다 — 이 경로의 목적이 **사용자가 물을 줄 모르는**(교차 링크가 안 보이는) 클러스터를 드러내는 것이라 보이지 않는 것은 물을 수 없기 때문이다(발견은 읽기전용이므로 기본값은 recall — 후보를 띄우고 LLM 이 판단하며, 정밀도는 엣지 커밋 단계인 Phase 1 `불확실→신규`·승격 게이트가 맡는다). 한 번의 도구 호출로 전 카테고리의 자매 마커를 평면 재료화한다:
   - `grep -rEn '자매\([^)]*\)' docs/compound/` — 전 카테고리를 재귀로 훑어 `자매(기준)` 마커가 박힌 사례 줄을 `path:line:사례 줄 전체` 형태로 평면 출력한다(`-rEn`: 재귀+정규식+줄번호이며 `-o` 가 없어 토큰만이 아니라 줄 전체가 나온다 — 줄에 박힌 `자매(기준)` 토큰과 증상 맥락을 path 와 함께 띄워 LLM 이 클래스 동일성을 판단하게 하려는 의도다. 정규식은 size-on-read ③·curate step-a③ 의 `자매\([^)]*\)` 와 동일 — 평면 `grep -c '자매('` 처럼 세기만 하지 않고 사례 줄을 통째로 띄워 서로 다른 클래스를 한 덩어리로 합산하지 않는다). `docs/compound/` 가 없는 신규 프로젝트는 매치 대상이 없으므로 graceful skip(부재 경로엔 grep 이 에러를 낼 수 있으나 그 부재 자체를 결함으로 보고하지 않는다).
   - 오케스트레이터(LLM)가 이 평면 목록을 **한 방 관련성 판단**으로 읽어, **≥2 카테고리 path 에 걸친 동일 앵커**(+앵커 드리프트로 다른 이름이 붙었으나 같은 클래스인 것)를 교차-카테고리 클러스터 후보로 식별한다(grep 은 `## _superseded` 섹션에 격리된 자매 줄도 함께 잡으므로, path·줄 맥락으로 활성/격리를 분별해 격리분을 활성 클러스터로 오판하지 않는다). **기계 자동 군집 금지** — 앵커 키가 불안정(한 클래스가 4~5 이름)해 기계 그룹핑은 과소(드리프트 미병합)·과대(동음이의 오병합) 판정을 부르므로, 군집 판단은 LLM 의 평면 목록 1회 검토로만 한다. 다홉 그래프 워크가 아니라 단일 재료화→단일 판단이다.
   - **발견된 교차 클러스터의 행동(공유 원칙 승격 여부·승격 시 홈 카테고리 결정)은 오케스트레이터 메타결정이다** — `compound` 스킬 `## Gap 유형 → docs/compound 카테고리 매핑표` 의 권한 경계(신 카테고리 신설·매핑표 1행 추가가 오케스트레이터 직접 권한)와 동일 소유권이다. compound-learner 의 `compound-curate` 는 per-카테고리 유지(무손실 per-file 머신 불변)이므로 교차-카테고리 인지로 확장하지 않는다. 이 재료화는 **판단용 리포트일 뿐** 자동 `자매` 엣지 생성·자동 승격을 하지 않는다(거짓 엣지→거짓 클러스터→모두가 로드하는 공유 요약 오염을 막는 precision-over-recall 엣지).
3. (재발도, 계보밀도) 내림차순으로 정렬한 표를 **채팅에 텍스트로만** 출력하고, 2.5 의 교차-카테고리 클러스터 후보 목록을 별도 표로 병기한 뒤(≥2 카테고리에 걸친 앵커가 없으면 "교차-카테고리 클러스터 없음" 1줄), 종료한다. 숫자는 `compound` 스킬이 누적한 per-case 마커(`재발(`·`자매(`)에서 그 자리에서 파생되므로 별도 상태 파일을 유지할 필요가 없다.

> 이유: 자동 랭킹으로 "나쁜 파일 먼저 정리" 같은 행동을 트리거하려던 설계는 기각됐다 — 드레인(curate)은 한 번 호출되면 맡은 플래그 목록을 full 완료하므로 처리 순서가 종료 상태를 1바이트도 바꾸지 못한다(순서매김 공허). 살아남는 정당분은 사용자가 물을 때 그 자리에서 숫자를 파생해 보여주는 값싼 조회뿐이다. 재발도 키는 합성-전용일 수 있으므로(`재발(`=0 인 코퍼스에선 자매 집중만 live), 이 표는 진단 보조이지 자동 행동의 근거가 아니다. compound 스킬의 `자매(`·`재발(` 마커(B 트랙)를 읽으므로 그 마커 도입 뒤에 의미가 산다. 2.5 의 교차-카테고리 발견을 이 COLD 경로(건강 질의)에만 둔 것도 같은 경제 논리다 — 매 버그 작업 핫패스(오케스트레이터가 통째 주입하는 `{카테고리}/README.md`·작업 봉투)에 카테고리 횡단 재료화를 넣으면 무세금 핫패스가 매번 그래프 인지 비용을 무는데, 교차 자매는 curate·건강 질의 같은 저빈도 COLD 일거리라 사용자가 물을 때만 1회 재료화하면 충분하다. 핫패스는 무탐색·평면 1홉 push 를 유지하고, 그래프 인지(전 카테고리 마커 횡단)는 COLD 의 기계적 materialize-then-show 로 격리한다.

## 계획 정합성 검증 요청 시 처리 경로

1. 오케스트레이터 파이프라인(Phase 1~5)을 실행하지 않는다.
2. `plan-auditor` 에이전트를 직접 호출한다 — 감사 대상 마일스톤 번호(또는 "전체")와, 관련 정본 경로(체크리스트·작업 로그·근거 audit 보고서)를 전달한다. 사용자 응답이 필요하므로 백그라운드 실행이 아니라 동기 호출로 진행한다.
3. plan-auditor 반환 보고서(`_workspace/plan_audit_*.md`)의 결함 목록과 3등급 분류(결함 아님 / 문서 수정 / 재판단 필요)를 사용자에게 요약 보고한다.
4. 사용자가 정본 수정을 승인하면 `milestone-tracker` 에이전트에 구체적 수정 지시를 위임한다. plan-auditor 는 정본을 직접 수정하지 않는다 — `milestone-status.md` 는 milestone-tracker 단일 관할이다.
5. 다음 스텝 제안과 함께 재개 프롬프트를 동반 출력하고 종료한다.

> 이유: 계획 정합성 감사는 마일스톤 파이프라인과 무관한 진단 작업이다. 코드↔스펙 검증(evaluator)이나 체크리스트 갱신(milestone-tracker)과 달리, 계획 문서들끼리의 수평 정합성(drift·staleness·중복·분류 오류·진술 오류)을 본다. 정본 수정은 milestone-tracker에 위임하여 단일 관할을 유지한다(DRY).
