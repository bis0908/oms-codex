> **구현 에이전트 주의.** 이 하네스는 검증·학습·추적·조율·기록 역할(qa-guard, security-auditor, evaluator, design-reviewer, milestone-tracker, session-archivist, plan-auditor, compound-learner, tdd-agent, refactor-specialist, bug-fixer)과 기본 구현 역할(`page-builder`, `data-layer`)을 포함한다. 아래에서 "프론트 구현 에이전트"는 `page-builder`, "백엔드 구현 에이전트"는 `data-layer`를 기본으로 가리킨다. 프로젝트가 별도 구현 에이전트를 정의했다면 프로젝트 `AGENTS.md`의 override를 우선한다. 레거시 분석 전용 역할은 기본 하네스에 포함하지 않는다.

> **기본 구현 에이전트 선택.** 오케스트레이터는 선언이 없으면 아래 기본값을 사용한다. 프로젝트 `AGENTS.md`가 별도 구현 에이전트 이름을 선언하면 그 값을 우선한다.
>
> | 구현 영역 | 기본 에이전트 | 프로젝트 override |
> |---|---|---|
> | 프론트 구현 | `page-builder` | (프로젝트가 선언하면 우선) |
> | 백엔드/API/데이터 구현 | `data-layer` | (프로젝트가 선언하면 우선) |

## Phase 0: 컨텍스트 확인

작업 시작 전 현재 상태를 파악한다.

### 공통 실행 계약

이 스킬은 전체 파이프라인의 기준 규칙이다. 각 에이전트 문서는 역할과 입출력 책임만 정의하고, 세부 실행 절차는 해당 스킬을 우선한다. 반환 형식이 에이전트 문서·스킬 양쪽에 정의되면 스킬 정의를 정본으로 한다.

모든 하위 에이전트 호출에는 아래 작업 봉투를 포함한다.

```markdown
## 작업 봉투

- 목표: <이번 호출에서 완료할 사용자 가치 1문장>
- 범위: <구현·검증 대상 파일 또는 기능 목록>
- 제외: <구현하지 않을 항목>
- 참조: <티켓, 원본 스펙, 디자인 레퍼런스, API 명세>
- 완료 기준: <검증 가능한 조건>
- 금지 사항: <이번 호출에서 절대 하지 않을 작업>
- 반환 형식: <표준 반환 형식>
```

작업 봉투에서 `목표`, `범위`, `완료 기준` 중 하나라도 비어 있으면 구현 에이전트를 호출하지 않는다. 오케스트레이터가 먼저 티켓 또는 문서를 다시 읽어 값을 채운다.

### 재발 방지 학습 푸시 (의무)

서브 에이전트가 SKILL.md 포인터를 스스로 따른다는 보장이 없으므로, 오케스트레이터가 강제 주입한다. 아래 에이전트를 호출할 때는 작업 봉투 `참조` 에 해당 카테고리 인덱스 경로를 **반드시 포함**하고, 봉투 본문에 다음 지시를 넣는다:

> 작업 시작 전 `docs/compound/{카테고리}/README.md` 를 읽고, 각 유형 파일 첫 줄 load-when 술어를 확인하여 현재 작업이 해당하는 사례 파일만 로드해 같은 실수를 반복하지 마라.

| 호출 에이전트 | 주입할 카테고리 인덱스 |
|---|---|
| 프론트 구현 에이전트 | `docs/compound/frontend/README.md` |
| 백엔드 구현 에이전트 | `docs/compound/data-layer/README.md` |
| evaluator | `docs/compound/evaluation/README.md` |
| qa-guard | `docs/compound/qa/README.md` |
| security-auditor | `docs/compound/security/README.md` |
| design-reviewer | `docs/compound/design/README.md` |
| bug-fixer | `docs/compound/bugfix/README.md` |
| tdd-agent | `docs/compound/tdd/README.md` |

`docs/compound/` 가 아직 없는 신규 프로젝트는 주입할 인덱스가 없으므로 이 단계를 건너뛴다 — compound-learner가 첫 학습 시 카테고리를 생성한다. 이 주입을 누락하면 자가개선 루프의 소비 단계가 끊긴다. 배경은 `compound` 스킬의 `## 목적` 참조.

**harness 카테고리 자기 소비 (오케스트레이터 직접 읽기).** 위 표에는 `harness` 카테고리가 없다 — 그 사례(게이팅 화이트리스트·라우팅 휴리스틱·권한 경계 등 하네스 운영 결정)를 소비할 서브 에이전트가 없기 때문이며, 그래서 harness는 소비자 없이 적재만 되는 write-only가 된다. **harness 학습의 소비자는 오케스트레이터 자신이다**(카테고리 표의 신규 카테고리 신설·메타 결정 소유자가 오케스트레이터인 것과 동일 논리). 따라서 오케스트레이터는 라우팅·게이팅 결정을 적용하기 전에(`입력 유형 분기`·버그 경로 evaluator 게이팅·Phase 4 증분 재게이팅·5-D 위험 클래스 게이트) `docs/compound/harness/README.md` 가 있으면 1회 읽어 과거에 누적된 하네스 메타 결함을 같은 자리에서 반복하지 않는다. **파일/디렉토리가 없으면 graceful skip**(위 신규 프로젝트 건너뜀과 동일 — 부재를 결함으로 보고하지 않는다). 이는 위 "재발 방지 학습 푸시" 표(서브 에이전트 작업 봉투 주입)와 **다른 메커니즘**이다 — 서브 에이전트 주입이 아니라 오케스트레이터 자기 읽기이므로, 표에 harness 행을 추가해 존재하지 않는 서브 에이전트에 주입하려 하지 말 것(표와 충돌하지 않는다).

**size-on-read — curate-needed 플래그 적재 (작업 차단 없음).** 위 표대로 카테고리 `README.md` 를 작업 봉투 `참조` 에 주입할 때, 그 카테고리의 유형 파일이 아래 **기계 트리거** 중 하나라도 넘으면 해당 경로를 사유 태그와 함께 **`_workspace/curate-needed.md` 에 1줄 append** 한다(세션 메모 같은 인컨텍스트 상태가 아니라 파일로 외부화하여 compaction 후에도 보존되게 한다 — 현재 작업 흐름은 막지 않는다, 읽기 비용이 곧 경제 신호다). 모든 트리거는 **줄 수·바이트·토큰 카운트 같은 기계 판정으로만** 수행한다 — "이 사례가 중요한가" 같은 의미 판단은 금지.

- **① 줄 수**: `{유형}.md` 가 임계(기본 **150줄**, 튜닝 가능) 이상 → `'{유형}.md 사유:줄수'`.
- **② README 행 바이트**: 카테고리 README 에서 그 `{유형}.md` 인덱스 행 1줄의 바이트가 임계(기본 **4KB**)를 초과 → `'{유형}.md 사유:README행 비대'`. 사례 파일은 사례 1건이 매우 긴 단일 bullet 로 누적돼 줄 수 신호가 약할 수 있으므로, supersede 체인 누적으로 한 README 행이 비대해진 신호를 줄 수와 독립으로 잡는다(행 단위 바이트로만 판정, 의미 판단 금지).
- **③ 자매 카운트 (클래스별)**: `{유형}.md` 에서 `자매({기준})` 토큰을 기준별로 그룹 카운트(`grep -oE '자매\([^)]*\)' {유형}.md | sort | uniq -c`)하여 **한 기준의 자매가 N(기본 3) 이상**이면 → `'{유형}.md 사유:자매 N건({기준})'`. 이것이 자매 마커의 진짜 소비자 결선이다 — 자매가 쌓여도 <150줄·<4KB행인 파일은 ①②로는 영영 안 잡힌다. 평면 `grep -c '자매('` 금지(서로 다른 클래스를 합산해 오판). **N(기본 3)은 미검증 튜닝값**이다: 동결 코퍼스에서 `재발(`=0 이고 자매 마커는 실데이터 검증이 전무하므로, 실사례 누적과 curate step-b 앵커 정규화(`compound` 스킬 `### 정리 트랙`)가 선 자리잡은 뒤 조정한다(그 전엔 한 클래스가 여러 이름으로 불려 과소카운트=미발동으로 보수적으로 빠진다 — 거짓 발동보다 안전).

이 플래그는 Phase 5 드레인에서 소비된다.

**표준 반환 형식**

모든 구현·검증 에이전트는 아래 키를 유지한다. 해당 없는 항목은 `없음`으로 작성한다.

```markdown
완료 항목:
- <파일 또는 검증 항목>

미완료 항목:
- <없음 또는 항목>

확인 필요:
- <없음 또는 질문>

다음 단계:
- <오케스트레이터가 수행할 작업>

마일스톤: M{N}
단계: <phase>
에이전트: <agent-name>
```

### 입력 유형 분기

**가장 먼저** 사용자 입력이 다음 중 어느 유형인지 판별한다:

| 입력 유형 | 판별 기준 | 처리 경로 |
|-----------|----------|-----------|
| 마일스톤/기능 구현 요청 | "M3 작업", "사용자 목록 페이지 구현" 등 | Phase 0 나머지 → Phase 1~5 정상 파이프라인 |
| 리팩토링/구조 개선 요청 | "중복 제거", "함수 분리", "파일 정리", "이름 변경", "구조 개선" 등 | Phase R → `refactor-specialist 에이전트 호출` → `qa-guard 에이전트 호출` |
| **버그 증상 보고** | 에러 메시지, "안 됨", "깨짐", "안 보임", 스택트레이스 포함 | 아래 버그 경로로 즉시 이탈 |
| **재개 프롬프트 요청** | "다음 세션 프롬프트", "재개 프롬프트", "이어서 할 프롬프트", "handoff 프롬프트 만들어줘" 등 | 아래 재개 프롬프트 경로로 즉시 이탈 |
| **세션 기록/인계 보존 요청** | "히스토리 저장", "인계 문서", "다음 세션에서 불러오게", "작업 기록 남겨줘" 등 | 아래 세션 기록 경로로 즉시 이탈 |
| **compound 정리/정돈 요청** | "compound 정리해줘", "재발방지 사례 정돈", "누적 학습 정리" 등 (버그 수정·기능 구현 아님 — 정리를 **실행**하는 쓰기/유지보수 동작) | 아래 compound-curate 온디맨드 경로로 즉시 이탈 |
| **compound 건강 질의** | "어느 사례가 가장 자주 재발하나", "자매가 어디 몰렸나", "재발/군집 집중도 보여줘", "학습 건강 현황", "여러 카테고리에 걸친 같은 클래스 있나"(교차-카테고리 자매 조회) 등 (정리 실행이 아니라 숫자를 **조회·보고**하는 읽기 동작) | 아래 compound 건강 질의 온디맨드 경로로 즉시 이탈 (교차-카테고리 자매 재료화 포함) |
| **계획 정합성 검증 요청** | "정합성 확인", "체크리스트 아직 유효한지", "계획 점검", "next step 계획 맞는지", "마일스톤 계획 검증" 등 (구현·버그 아님) | 아래 계획 정합성 감사 경로로 즉시 이탈 |

**버그 증상 보고 시 처리 경로:**
1. 오케스트레이터 파이프라인(Phase 1~5)을 실행하지 않는다
2. `bug-fixer` 에이전트를 직접 호출하여 버그를 수정한다 (작업 봉투 `참조`에 `docs/compound/bugfix/README.md` 의무 포함 — 카테고리가 있으면. "재발 방지 학습 푸시" 표 참조)
3. **evaluator 게이팅 — 동작 레이어 경로-prefix 화이트리스트 판정.** bug-fixer 반환 메시지 `완료 항목`의 파일 목록을 추출하여, 프로젝트가 정의한 동작 레이어 화이트리스트 prefix 중 **하나라도 닿는지**만 본다. 판정은 경로 prefix 매칭 한 줄로만 수행한다 — "비즈니스 로직을 건드렸는가" 같은 의미 판단은 금지. 평가 깊이는 **skip / full 이진**이며 경량 evaluator 모드는 없다. 화이트리스트가 프로젝트에 미정의면 보수적으로 전 파일을 **닿음**으로 취급해 full evaluator 를 실행한다(증분 재게이팅의 미정의 기본값과 동일 — 보안 게이트는 graceful skip 대상이 아니다, 미게이팅보다 과게이팅이 싸다).

   ```
   동작 레이어 화이트리스트 (prefix) — 프로젝트가 정의. 일반 분류:
     데이터 접근 레이어   서비스 레이어   DB 접근   API 핸들러
     인증   세션
   (경로 기준은 프로젝트 AGENTS.md 또는 init-project가 남긴 후보를 참조)
   ```

   - **닿음** (완료 항목 중 위 분류에 매칭되는 파일이 1개 이상): `evaluator` 에이전트를 호출하여 수정 결과가 원본 스펙을 충족하는지 검증한다 — bug-fixer 반환 메시지의 `완료 항목` 파일 목록과 수정 맥락(버그 증상, 원인)을 함께 전달한다 (작업 봉투 `참조`에 `docs/compound/evaluation/README.md` 의무 포함)
     - 보고서 파일명은 `_workspace/eval_bugfix_{증상-슬러그}_{datetime}.md` 로 지정해 봉투에 명시한다 (마일스톤 게이트의 절대 라운드 상한이 `eval_{마일스톤}_*` 파일 개수를 기계 카운트하므로 — `references/error-handling.md` 정본 — 버그 경로 보고서가 같은 네임스페이스를 오염시켜 회차를 부풀리지 않게 분리한다)
     - evaluator 승인 → 4번으로 진행
     - evaluator 미승인 → bug-fixer에 재수정 요청 → evaluator 재검증. 동일 미해결 항목이 2회 연속이면 stuck 판정 후 에스컬레이션 (버그 경로 전용 2회 기준 — 단발 경로라 빠른 반환이 저렴. 파이프라인 3회와 다름. 임계·동일성 키는 `references/error-handling.md` 정본 참조)
   - **안 닿음** (완료 항목이 스타일 파일·컴포넌트 마크업·className/JSX-only 등 동작 레이어 외에만 한정): evaluator를 **스킵**한다. bugfix 스킬 Step 4.5 자가 검증 + 사용자 시각 검증으로 충분하며, 무거운 3자 대조·브라우저 스모크 라운드트립을 생략한다. 곧바로 4번으로 진행한다.
   - **위험 주석**: className 리네임에 조건부 렌더 누락이 묻어드는 케이스가 있다 — 그래서 화이트리스트를 **엄격 적용**(prefix 매칭 외 임의 확대 금지)하여 동작 레이어가 닿으면 반드시 full evaluator로 잡는다.
4. **compound-learner 호출 게이트 — 사용자 시각 검증 분기.** evaluator 게이팅 결과에 따라 분기한다.
   - **evaluator 닿음 + approved**: evaluator가 동작 정합성을 보증했으므로 먼저 bug-fixer 반환 `완료 항목` 파일만 `git add` 로 선스테이징한 뒤 `commit` 스킬로 커밋한다 — 메시지 `fix: {버그 증상 한 줄 요약}` (선스테이징은 commit 스킬이 전량 스테이징하는 것을 막는 결정적 절차다. 버그 경로엔 마일스톤 커밋 체크포인트가 없어 검증된 수정이 세션 크래시에 휘발되며, 범위 한정은 진행 중 마일스톤 작업분이 섞이는 것을 막는다). 커밋 실패 시 사용자에게 알리고 수동 커밋을 안내한다. 커밋 직후 `compound-learner` 에이전트를 백그라운드 학습으로 호출한다(인자: 트리거 `bug-fix`, 보고서 `bug-fixer` 반환 메시지 전문). 커밋 → 백그라운드 학습 순서는 5-C와 동일하며, 학습 산출물이 커밋에 섞이는 경합을 피한다.
   - **evaluator 안 닿음** (스타일/마크업 단독, evaluator 스킵 케이스): compound-learner를 **즉시 호출하지 않는다**. 5번으로 진행해 사용자에게 시각 검증을 요청하고, 6번에서 사용자가 "확인됐다" 긍정 응답을 한 경우에만 compound-learner를 호출한다.
   - **이유**: 시각 결함은 사용자 시야가 단독 정보 소스다. bug-fixer 자체 검증은 코드 차원이라 "박스가 사라졌는지"를 확정할 수 없다. evaluator 스킵으로 비워진 검증 자리를 사용자 응답이 메우지 않으면, 잘못된 학습(1차 오진단을 정답으로 누적)이 영구 기록된다.
5. 사용자에게 bug-fixer 수정 내용 + evaluator 결과(또는 스킵)를 전달한다.
   - **게이팅 결정 로깅(의무)**: 보고에 `evaluator 게이팅: 닿음 — 매칭 경로 {prefix/파일}` 또는 `evaluator 게이팅: 안 닿음 — 동작 레이어 외 (스킵)` 를 **1줄 명시**한다. 회귀 가림을 사후 감사할 수 있도록 누락 금지.
   - **evaluator 닿음 경로**: 보고에 커밋 해시를 1줄 포함하고 그대로 종료한다(커밋과 compound-learner 백그라운드 호출은 4번에서 이미 수행됨).
   - **evaluator 안 닿음 경로**: 종료하지 않고, "이 수정으로 시각 결함이 실제로 해결됐는지 브라우저에서 확인 부탁드립니다" 를 명시한 뒤 사용자 응답을 대기한다. **재발 방지 학습은 사용자 응답 후 6번에서 수행됨**도 함께 명시한다.
6. **사용자 시각 검증 응답 처리 (evaluator 안 닿음 경로에서만 실행).** 다음 턴에서 사용자가 보내는 응답에 따라 분기한다.
   - **긍정 응답** ("확인됐다", "해결됐다", "잘 됨", "오케이", "박스 사라짐" 등): 먼저 bug-fixer 반환 `완료 항목` 파일만 `git add` 로 선스테이징한 뒤 `commit` 스킬로 커밋한다 — 메시지 `fix: {버그 증상 한 줄 요약}`, 커밋 실패 시 수동 커밋 안내(선스테이징은 전량 스테이징 방지). 커밋은 반드시 긍정 응답 뒤에만 한다(시각 검증 전 커밋은 미검증 수정을 체크포인트로 굳힌다 — evaluator 스킵 자리를 메우는 검증이 사용자 응답이기 때문). 커밋 직후 `compound-learner` 에이전트를 백그라운드 학습으로 호출한다(인자: 트리거 `bug-fix`, 보고서 `bug-fixer` 반환 메시지 전문 + 사용자 시각 검증 결과). 커밋 해시를 1줄 보고한 뒤 종료한다. **이때 봉투에 "사용자 시각 검증 통과(응답 인용)"를 명시**하여 정답으로 누적되도록 한다. 호출 후 `_workspace/bugfix-attempts.md` 가 존재하면 비운다(truncate — 재수정 1회 종결 케이스 포함, 잔존 시 다음 버그 가설에 섞인다).
   - **부정/미해결 응답** ("수정 안됨", "여전히 보임", "다른 곳도 동일" 등): compound-learner를 **호출하지 않는다**. 2번 bug-fixer 재호출 트랙으로 회귀하여 추가 진단·수정한다. 사용자 응답 본문(스크린샷·위치 추가 정보)을 새 bug-fixer 봉투의 `진단 맥락`에 포함한다. 재호출 전에 버그 식별(증상 한 줄)·회차·직전 가설(원인 1줄)을 `_workspace/bugfix-attempts.md` 에 1줄 append 한다 — 다회 재수정 가설 이력의 외부 기억(compaction 후에도 보존). 파일에 다른 버그의 잔존 줄이 있으면 먼저 비운다. 재수정 후 다시 5번으로 진입한다.
   - **2회 이상 재수정이 누적된 경우**: 최종 긍정 응답 후 compound-learner 호출 시, 1차·2차 가설과 정진단 흐름을 `_workspace/bugfix-attempts.md` 에서 읽어 봉투에 함께 전달하여 compound-learner가 자매 보존(둘 다 가치 있음) vs `_superseded` 격리(1차가 오진단으로 가치 없음) 를 판단하게 한다. 봉투에 "1차는 오진단이었다 — 격리 검토 권고" 또는 "1차는 a11y 보강으로 가치 보존" 의 판단 근거를 명시한다. 호출 직후 이 파일을 비운다(truncate — curate-needed 와 동일 패턴, 비우지 않으면 다음 버그의 가설 이력에 섞인다).

> 이유: 버그 수정은 마일스톤 파이프라인과 무관한 단발성 작업이다. design-reviewer 등 불필요한 에이전트를 호출하지 않는다. evaluator는 고비용 3자 대조와 브라우저 스모크를 수행하는 무거운 검증이므로, 동작 레이어가 닿을 때만 실행하고 순수 시각/마크업 수정에는 생략하여 개입을 경제화한다. compound-learner 학습은 evaluator 닿음 케이스는 evaluator approved 가 검증을 대체하고, 안 닿음 케이스는 사용자 시각 검증이 그 자리를 메운다 — 두 경로 모두 검증된 정답만 누적되도록 게이트를 둔다.

> **예외 — 신규 인증/세션 설계가 섞인 버그 경로**: "버그 수정" 봉투로 들어왔어도 그 해결이 단순 코드 교정이 아니라 **신규 토큰·쿠키·세션 메커니즘 설계**(예: 임퍼소네이션 별도 쿠키, 자동로그인 재설계)를 요구하면 bug-fixer 단독으로 처리하지 않는다. ① 세션/쿠키 메커니즘(별도 쿠키 vs 기존 세션 변형, TTL, 서명 방식)을 사용자에게 1회 선결정 받고, ② `tdd-agent` 에이전트 → 백엔드 구현 에이전트 → 프론트 구현 에이전트 트랙으로 분리 호출한 뒤 evaluator·qa-guard·design-reviewer로 검증한다. 판별 기준: 수정이 새 쿠키명·새 토큰 발급/검증 함수·인증 분기 신설을 동반하면 단발 버그가 아니라 설계 작업이다.

**재개 프롬프트 요청 시 처리 경로:**

다음 세션이 작업을 정확히 이어받도록, 정본을 읽어 붙여넣기용 프롬프트 **텍스트만** 출력한다.

1. 파이프라인(Phase 1~5)·에이전트를 호출하지 않는다. **파일을 생성하지 않는다** (정본은 `docs/progress/milestone-status.md`).
2. 아래 정본을 직접 읽어 값을 채운다.
   - `docs/progress/milestone-status.md` 요약 섹션 — 현재 마일스톤·진행 단계·다음 작업·미완료 항목
   - 진행 중 마일스톤 섹션의 미완료(`[ ]`) 티켓, 배치 분할이 있으면 다음 배치 티켓 목록
   - `git branch --show-current`, `git log --oneline -1` — 브랜치·마지막 커밋 해시
   - `git status --short` — 미커밋 잔여 변경(있으면 프롬프트에 경고로 명시)
   - 최신 `_workspace/eval_*` / `_workspace/qa_*` / 원본 스펙 파일 경로 중 다음 작업이 참조할 것
   - 진행 중 보완 루프가 있으면 그 상태를 1줄로 명시 — 게이트명(qa-guard/evaluator/design-reviewer) + 직전 미해결 항목 + 회차. 값은 최신 `_workspace/qa_*`·`_workspace/eval_*` 보고서 머리말 `반복 횟수: {N}회차 / 직전 미해결: {목록}`(design-reviewer 게이트는 회차를 `_workspace/design_review_{해당 scope}_*` 보고서 파일 개수로 세고 — 머리말 `반복 횟수: {N}/5` 는 단일 호출 내 내부 수정 루프 카운터라 외부 보완 회차가 아니다 — 미해결 항목은 최신 보고서 FAIL/미완료 항목에서 읽는다)에서 읽어 채운다.
3. **새 세션 재개 프롬프트**(아래 "컨텍스트 잔여량 감시 루프" 섹션의 동일 포맷)를 재사용하되, 프로젝트 컨텍스트로 보강한다.
   - 첫 줄은 `$orchestrate <다음 작업 한 줄>`로 시작 — 새 세션이 스킬을 바로 진입하도록
   - 브랜치명·마지막 커밋 해시, 다음 배치/티켓 목록, 다음 작업이 먼저 읽어야 할 정본·`_workspace` 파일 경로, 미결 정리 항목(미커밋 변경·dev DB 잔여·기술부채 메모)을 포함
   - 직전에 확정됐으나 정본에 미반영된 설계 결정이 있으면 프롬프트에 1줄로 명시
4. 프롬프트를 단일 ```text 코드 블록으로 출력하고, 정본 파일 경로를 한 줄로 안내한 뒤 종료한다.

> 이유: 재개 프롬프트 생성은 마일스톤 파이프라인과 무관한 메타 작업이다. 별도 핸드오프 문서를 만들면 정본(milestone-status.md)과 상태가 분산되므로(DRY) 텍스트만 합성한다.

**세션 기록/인계 보존 요청 시 처리 경로:**

사용자가 세션 기록 또는 인계 문서 보존을 직접 요청하면 마일스톤 파이프라인에 진입하지 않고 `session-archivist`만 호출한다.

1. `docs/progress/**`와 최신 `_workspace/qa_*`·`_workspace/security_*`·`_workspace/eval_*`·`_workspace/plan_audit_*`를 읽어 사실 집합을 만든다.
2. `session-archivist` 에이전트를 호출한다. 작업 봉투에는 요청 문장, 완료 항목, 변경 파일, 검증 결과, 다음 세션 첫 작업을 포함한다.
3. 직접 재개 프롬프트 요청과 섞인 경우 우선순위를 분리한다.
   - "프롬프트만" 요청: 이 경로를 타지 않고 재개 프롬프트 경로를 유지한다.
   - "파일로 인계도 저장" 요청: 이 경로를 실행한 뒤 필요하면 프롬프트 텍스트를 별도 출력한다.
4. `session-archivist`는 `docs/history/**` 또는 `docs/handoffs/**`만 작성한다. `docs/progress/**`는 읽기 전용이다.

> 이유: 세션 로그와 인계 문서는 장기 보존 산출물이고, 재개 프롬프트는 붙여넣기용 임시 텍스트다. 둘을 같은 경로로 합치면 정본과 보조 산출물의 경계가 흐려진다.

**compound-curate 온디맨드 / compound 건강 질의 / 계획 정합성 검증 요청 시 처리 경로:**

세 경로 모두 파이프라인에 진입하지 않는 저빈도 비파이프라인 진입이다. 입력이 위 분기 표에서 이 세 유형 중 하나로 판별되면 `references/input-branch-paths.md` 의 해당 섹션을 읽어 그 절차를 따른다. (버그 경로·재개 프롬프트 경로는 빈출/결합 경로이므로 본문에 유지된다.)

### Git 브랜치 준비

새 마일스톤 시작 시 반드시 실행한다:

1. `git branch --show-current`로 현재 브랜치 확인
2. **✅↔커밋 크래시 윈도우 대조** — `git status --short` 를 1회 실행한다. 출력이 비어 있지 않은데 `docs/progress/milestone-status.md` 요약 표의 "진행 단계"가 `완료`이면(정본상 직전 마일스톤은 ✅인데 작업 트리에 미커밋 변경이 남음 — 5-C/5-D에서 tracker ✅ 직후·커밋 전에 세션이 끊겨 커밋이 누락된 신호), 새 브랜치를 만들지 말고 사용자에게 1회 확인한다: 잔여 변경을 직전 마일스톤 커밋으로 먼저 처리할지(`commit` 스킬, 메시지 `feat(M{직전 번호}): {요약}`, 실패 시 수동 커밋 안내) 폐기할지. 확인 전에는 3번으로 진행하지 않는다(더러운 트리 위에 새 브랜치를 만들면 직전 마일스톤 작업이 새 마일스톤 커밋에 섞인다). "진행 단계"가 `완료`가 아닌 미커밋 변경은 진행 중 마일스톤의 정상 작업분이므로 이 분기를 건너뛰고 계속하며, 정본 파일이 없는 신규 프로젝트도 건너뛴다.
3. 해당 마일스톤 브랜치(`feat/mN-{slug}`)가 없으면 `git checkout -b feat/mN-{slug} {기본 브랜치}`로 생성
4. 이미 존재하면 `git checkout feat/mN-{slug}`
5. 브랜치 전환 결과를 사용자에게 한 줄 보고 — 2번 대조 결과(`git 대조: 깨끗` / `git 대조: 미커밋 {N}건 — 확인 분기 발동` / `git 대조: 정본 부재 — 건너뜀`)를 함께 1줄 명시한다(게이팅 결정은 사후 감사를 위해 누락 금지)

브랜치 명명 예시: `feat/m0-bootstrap`, `feat/m1-db-layer`, `feat/m2-design-system`

### 마일스톤 진행 상황 복구

`docs/progress/milestone-status.md`가 존재하면 `milestone-tracker` 에이전트를 호출하여 세션 시작 상태 보고를 요청한다. 파일이 없으면(신규 프로젝트) 이 단계를 건너뛰고, 첫 마일스톤 착수 시 milestone-tracker가 생성한다.

tracker가 반환한 요약을 사용자에게 표시하고, 이어서 진행할지 새로 시작할지 확인한다.

### 컨텍스트 잔여량 감시 루프

각 Phase 시작 전과, tdd-agent / 구현 에이전트 / design-reviewer / qa-guard / security-auditor / evaluator / plan-auditor / session-archivist 반환 직후에 컨텍스트 잔여량을 보수적으로 추정한다.

**1차 게이트 — 정량 임계 (이것이 유일한 정상 중단 트리거다):**

- 컨텍스트 윈도우는 150k 토큰으로 본다. 마진 10%를 적용해 **누적 사용량이 약 135k 토큰(잔여 ~15k, 10%)에 도달했을 때만** 재개 프롬프트를 동반한 안전 중단을 발동한다.
- 정확한 토큰값을 직접 읽을 수 없으므로, 다음 프록시를 135k 기준으로 보수적으로 환산해 추정한다. **개별 조건이 아니라 누적 사용량이 임계에 닿았는지로만 판단한다.**
  - 자동 컨텍스트 요약(compaction)이 이미 1회 이상 발생했다 → 거의 확실히 임계 근접.
  - **compaction 감지 시 루프 제어 상태 재앵커**: 자동 요약이 1회 이상 감지되면, 정본(`docs/progress/milestone-status.md`)과 최신 `_workspace/qa_*`·`_workspace/eval_*` 보고서 머리말 `반복 횟수: {N}회차 / 직전 미해결: {목록}`(design-reviewer 게이트는 회차를 `_workspace/design_review_{해당 scope}_*` 보고서 파일 개수로 세고 — 머리말 `{N}/5` 는 단일 호출 내 내부 수정 루프 카운터다 — 미해결은 최신 보고서 FAIL/미완료 항목에서 읽는다)에서 현재 Phase·보완 루프 회차 카운터를 다시 읽어 재앵커한 뒤 판단을 이어간다 — 휘발된 인컨텍스트 상태를 외부 기억으로 복원한다. 버그 경로 재수정 회차·가설 이력은 `_workspace/bugfix-attempts.md` 에서 읽는다. 응답 대기 상태(5-B·5-D step 0·버그 경로 5번)는 외부 기록이 없으므로 직전 검증 요청 보고와 사용자 응답 유무로 판단하고, 모호하면 사용자에게 1회 확인한다.
  - 누적으로 연 파일·반환받은 에이전트 산출물의 총량이 길고, 직전 몇 턴 동안 응답·도구 결과가 지속적으로 길었다.
  - 위 신호가 약하면 아직 임계가 아니다 — **남은 Phase 수·반복 실패 횟수만으로는 중단하지 않는다.**
- **임계 미달이면 작업을 계속한다.** 남은 Phase·보완 루프가 2단계 이상 남아 있어도, 동일 실패가 반복돼도, 그 자체로는 중단 사유가 아니다.

**2차 보조 — 진행 불가 판정 (정량 게이트와 별개):**

- stuck 에스컬레이션(동일 FAIL 3회 또는 절대 라운드 상한 통산 7회 초과 — `references/error-handling.md` 정본), 외부 차단(권한 거부·DB 불가) 등 **컨텍스트와 무관하게 이번 세션에서 진행 자체가 불가능**할 때만 별도로 중단한다. 이는 "컨텍스트가 찼다"가 아니라 "더 진행할 수 없다"는 다른 사유이며, 보고 시 그 사유를 명시한다.

- 위 1차 또는 2차 판정이 나오면 즉시 안전 경계에서 작업을 멈춘다.
  1. 현재까지 완료한 항목과 미완료 항목을 짧게 정리한다.
  2. 필요한 경우 `docs/progress/milestone-status.md`를 갱신한다.
  3. 새 세션으로 이어서 작업하라고 권유한다.
  4. 아래 재개 프롬프트를 함께 제시한다.

**새 세션 재개 프롬프트**
```text
`docs/progress/milestone-status.md`를 먼저 읽고,
현재 마일스톤 M{N}의 {현재 단계}부터 이어서 작업해줘.
마지막 완료 항목은 {파일 또는 티켓 목록}이고,
다음 작업은 {다음 작업 1개}이다.
```

**[강제] 다음 스텝 제안 시 재개 프롬프트 상시 동반**

오케스트레이터가 사용자에게 "다음 스텝 제안"을 출력하는 **모든 지점**에서, 그 제안 바로 다음에 위 **새 세션 재개 프롬프트**(단일 ```text 블록)를 **반드시** 함께 출력한다. 예외 없음.

- 적용 지점: Phase 5-C·5-D 완료 보고의 "다음 스텝 제안", 버그 경로 종료 지점, Phase R 리팩토링 완료 보고, 컨텍스트 감시 루프의 안전 중단, 그 외 다음 작업을 한 줄이라도 제안하는 모든 응답 종료부. (예외 1: 버그 경로 5번 evaluator 안 닿음 시 사용자 시각 검증 응답을 대기하는 단계 — 이때는 "응답 대기 자체"가 next-step이므로 별도 재개 프롬프트를 출력하지 않는다. 6번 긍정 응답 종료 시점에 동반 출력한다. 예외 2: Phase 5-B UI 마일스톤 사용자 검증 응답을 대기하는 단계 — 동일한 이유로 별도 재개 프롬프트를 출력하지 않는다. 5-C 긍정 응답 종료 시점에 동반 출력한다. 예외 3: Phase 5-D step 0 위험 클래스 게이트의 dev DB 적용 확인 응답을 대기하는 단계 — 동일한 이유로 별도 재개 프롬프트를 출력하지 않는다. 분기 후 5-D step 4 보고 시점에 동반 출력한다.)
- 채우는 값: "재개 프롬프트 요청 시 처리 경로" 2번과 동일한 정본을 직접 읽어 합성한다.
- 첫 줄은 `$orchestrate <다음 작업 한 줄>`로 시작하여 새 세션이 스킬에 바로 진입하도록 한다.
- 정본에 미반영된 직전 확정 설계 결정이 있으면 프롬프트에 1줄로 명시한다.
- 파일은 생성하지 않는다(정본은 milestone-status.md, DRY). 텍스트만 합성한다.

### _workspace 상태 분기

- `_workspace/` 존재 + 부분 수정 요청 → 해당 에이전트만 재호출
- `_workspace/` 존재 + 새 마일스톤 → 기존 `_workspace/`를 `_workspace_prev/`로 이동 후 새 실행. 이동 전에 `_workspace/curate-needed.md`·`_workspace/bugfix-attempts.md` 가 비어 있지 않으면 새 `_workspace/` 로 이월한다(드레인·소비 전에 회전으로 소실되는 것을 방지)
- `_workspace/` 없음 → 초기 실행 (첫 사용 시 생성)

## Phase R: 리팩토링/구조 개선

리팩토링 요청은 마일스톤 파이프라인과 분리한다.

1. 오케스트레이터가 변경 목적을 구조 개선으로 한정한다.
2. `refactor-specialist`에 대상 파일, 개선 목표, 보존 동작, 제외 범위를 전달한다.
3. 구조 변경만 완료되면 `qa-guard`로 보안·컨벤션·테스트를 확인한다. 보고서 파일명은 `_workspace/qa_refactor_{slug}_{datetime}.md` 로 봉투에 지정한다(마일스톤 게이트의 절대 라운드 상한 파일 카운트 비오염 — `references/error-handling.md` 카운트 비대상).
4. 동작 변경 가능성이 발견되면 `refactor-specialist`는 중단하고 상위 에이전트가 `bug-fixer` 또는 구현 에이전트로 재배치한다.
5. 요구사항 충족 변화가 의심될 때만 `evaluator`를 추가 호출한다.

## Phase 1: 마일스톤 분석

0. **계획 정합성 선점검 (마일스톤 착수 전 자동).** 요구사항 추출에 앞서 `plan-auditor` 에이전트를 호출하여, 착수할 마일스톤의 체크리스트가 근거 문서(로드맵·audit 보고서·작업 로그)와 정합한지 점검한다. 감사 대상 마일스톤 번호와 정본 경로를 전달한다. (근거 문서가 없는 신규 프로젝트는 plan-auditor가 "대조 불가 — 근거 부재"로 빠르게 통과하므로 개입 비용이 작다.)
   - `정합성 결과: consistent` → 그대로 1번으로 진행한다.
   - 결함 발견 → 사용자에게 결함 목록과 3등급 분류를 보고한다. 잘못된 계획으로 구현에 진입하지 않는다. "문서 수정" 결함은 milestone-tracker에 위임해 정본을 바로잡은 뒤 진행하고, "재판단 필요" 결함은 사용자 결정을 받은 뒤 진행한다.
1. 프로젝트 로드맵 문서에서 해당 마일스톤 요구사항 추출 (있을 경우)
2. 작업 티켓 문서에서 세부 작업 목록 확인
3. 작업 유형 분류:
   - 원본/레거시 참조 필요 여부 → 기존 원본 스펙·분석 문서 확인
   - 프론트 구현 필요 여부 → 프론트 구현 에이전트 투입 결정
   - DB/API 구현 필요 여부 → 백엔드 구현 에이전트 투입 결정
   - UI 구현 포함 여부 → **프론트 구현 에이전트가 투입되면 Phase 4에서 `design-reviewer` 자동 투입** (qa-guard 실행 전 선행)

> **마일스톤별 구현 에이전트 선택.** 어느 마일스톤에 어떤 구현 에이전트를 투입하는지는 로드맵·티켓·프로젝트 `AGENTS.md`를 근거로 정한다. 일반 규칙: ① UI가 포함되면 `page-builder` + Phase 4 design-reviewer, ② DB/API가 포함되면 `data-layer` + Phase 2.5 tdd-agent, ③ 헤드리스(결정 문서·마이그레이션 등 사용자 가시 UI 없음)면 Phase 5-D 즉시 커밋 경로.

## Phase 2: 원본 스펙 확인 (있을 경우)

원본/레거시 참조가 필요한 경우 이미 존재하는 원본 스펙, 분석 문서, API 명세, 기존 시스템 문서를 확인한다. 기본 하네스는 레거시 분석 전용 역할을 포함하지 않는다.

원본 스펙이 충분하면 해당 경로를 이후 작업 봉투 `참조`에 포함한다. 원본 스펙이 없고 `목표`, `범위`, `완료 기준`을 채울 수 없으면 구현 에이전트를 호출하지 않고 사용자에게 정본 제공 또는 별도 분석 작업을 요청한다.

이 Phase는 기본 산출물을 만들지 않는다. 사용자가 별도 분석 작업을 지시해 `_workspace/01_legacy_*.md` 같은 문서가 생성돼 있으면 그 문서를 원본 스펙으로 참조한다. 신규 개발 프로젝트(원본 스펙 없음)는 이 Phase를 건너뛴다.

### 범위 축소 결정의 사용자 확인 (필수)

포팅 중 원본 요소를 구현 범위에서 빼기로 결정할 때(원본 결함·범위 조정 등 사유 불문), 그 결정이 **사용자 가시 구조 단위** — 테이블 열, 행 액션 버튼, 탭, 페이지, 메뉴 항목, 화면 섹션 — 를 제거·축소하면, 사용자에게 묻는 질문에 **구체적 before→after 구조 델타**를 명시한다.

- 나쁜 예: "원본의 깨진 요소는 어떻게 할까요?" — 카테고리 라벨만. 사용자가 가시적 결과를 알 수 없다.
- 좋은 예: "선정안내·티켓 2개 열을 빼면 원본 12열 테이블이 10열이 됩니다. 빼는 데 동의하시나요?" — before→after 델타 명시.

이유: "백엔드 메서드가 깨졌다"와 "테이블에서 열이 사라진다"는 사용자 체감 크기가 다르다. 후자를 전자 질문에 동봉하면 가시적 영향이 부각되지 않아, 사용자가 완성물을 보고서야 차이를 인지하게 된다. 이렇게 확정된 제외 항목과 승인 출처는 원본 스펙 문서 또는 작업 봉투에 기록하여 Phase 4 evaluator 가 "의도적 제외 목록"으로 보고하도록 전달한다.

## Phase 2.5: TDD — 실패 테스트 작성 (백엔드 구현 에이전트 투입 시에만)

백엔드 구현 에이전트가 투입되는 마일스톤(데이터 접근·서비스·API 구현 단계)에서만 실행한다. 목업·결정 문서 단계는 이 단계를 건너뛴다.

`tdd-agent` 에이전트 호출 — 마일스톤 번호, 백엔드 구현 에이전트가 구현할 API/쿼리 목록, 원본 스펙 문서 경로(있을 경우)를 전달한다. (작업 봉투 `참조`에 `docs/compound/tdd/README.md` 의무 포함)

- 산출물: 데이터 접근·API 레이어의 실패 테스트 파일 (Red)
- tdd-agent는 테스트 작성 + Red 실행 확인 후 반환한다
- tdd-agent 반환 메시지의 "완료 항목"(테스트 파일 목록)을 Phase 3 백엔드 구현 에이전트 호출 시 함께 전달한다

프론트 구현 에이전트만 투입되는 마일스톤은 이 단계를 건너뛴다.

## Phase 3: 구현 (서브 에이전트 순차 호출)

실행 모드: 서브 에이전트. 각 에이전트는 격리된 컨텍스트에서 실행되며, 최종 반환 메시지만 오케스트레이터에 전달된다.

### Phase 3-0: 티켓 파싱 — 원자 작업 목록 생성 (에이전트 호출 전 필수)

에이전트를 호출하기 전에 오케스트레이터가 직접 작업을 분할한다. 이 단계를 건너뛰면 에이전트가 무엇을 누락했는지 확인할 수단이 없다.

1. 마일스톤 티켓 문서 읽기
2. **산출물** 섹션에서 모든 파일 경로 추출 (명시되지 않은 스타일·공용 컴포넌트도 포함)
3. 의존 관계 순서로 정렬:
   - 백엔드: 데이터 접근 레이어 → 서비스 레이어 → API 핸들러
   - 프론트: 공용/공유 컴포넌트 → 전용 컴포넌트 → 페이지
4. `현재 세션 작업 목록`로 각 파일을 **개별 태스크**로 등록하고 태스크 ID 목록 보관

### Phase 3-1: 에이전트 호출

에이전트 호출 프롬프트에 반드시 아래 내용을 포함한다:

```
## 작업 봉투

목표:
  M{N}의 <기능명>을 검증 가능한 상태로 구현한다.

범위:
  아래 구현 대상 파일 목록만 처리한다.

제외:
  티켓에 없는 새 기능, 리팩터링, 디자인 임의 변경, 운영 DB 접근

참조:
  - 마일스톤 티켓 문서
  - 원본 스펙 문서 (있을 경우)
  - 프로젝트 디자인 레퍼런스 (UI 작업인 경우)
  - docs/compound/{카테고리}/README.md (호출 에이전트별 — 카테고리가 있으면. 작업 전 필독·관련 유형 파일 로드)

완료 기준:
  - 모든 task_id가 completed 상태다.
  - 완료 항목에 실제 수정 파일을 전부 기재한다.
  - 목록에 없는 파일을 생성했으면 확인 필요에 사유를 남긴다.

구현 대상 파일 목록 (이 순서대로 처리할 것):
  [t001] ...
  [t002] ...

규칙:
- 각 파일을 구현한 즉시 현재 세션 작업 목록에서 해당 task_id를 `completed`로 갱신하라
- 현재 세션 작업 목록을 갱신하지 않은 채 다음 파일로 넘어가지 말 것
- 목록에 없는 파일을 추가 생성했다면 완료 보고 형식의 "확인 필요" 항목에 명시하라
- 표준 반환 형식을 유지하라
```

**모델 선택 규칙**

모든 하위 에이전트는 `model = "gpt-5.5"`로 호출한다. 하네스 품질은 에이전트 추론 능력에 직결되며, custom agent 정의 파일(`.codex/agents/*.toml`)도 전원 `model = "gpt-5.5"`로 통일돼 있다. 하위 에이전트 호출 시 `model = "gpt-5.5"` 파라미터를 명시한다.

**실행 경로**

> 백엔드 구현 에이전트 호출 시 Phase 2.5 tdd-agent가 생성한 테스트 파일 경로 목록을 함께 전달한다. 백엔드 구현 에이전트는 해당 테스트가 통과(Green)하도록 구현한다.

*API 의존 없음 — 병렬 실행:*

`백엔드 구현 에이전트 호출(인자: 백그라운드 실행, tdd 파일 목록 + 태스크 ID 목록 포함)` + `프론트 구현 에이전트 호출(인자: 백그라운드 실행, 태스크 ID 목록 포함)` 동시 호출 → 둘 다 완료 후 Phase 3-2 진행

*API 의존 있음 — 순차 실행:*

`백엔드 구현 에이전트 호출(인자: tdd 파일 목록 + 태스크 ID 목록 포함)` → Phase 3-2 검증 → 반환 메시지에서 구현된 API 명세 추출 → `프론트 구현 에이전트 호출(인자: API 명세 + 태스크 ID 목록 포함)` → Phase 3-2 검증

*프론트 구현 중 예상치 못한 API 필요 발견 시:*
1. 프론트 구현 에이전트가 `API 명세 필요` 반환 메시지로 구현 중단 (이미 완료된 파일 목록 포함)
2. 오케스트레이터가 백엔드 구현 에이전트에 해당 API 구현 + 관련 태스크 등록을 요청
3. 백엔드 구현 에이전트 반환 메시지(API 명세)와 함께 프론트 구현 에이전트를 재호출

### Phase 3-2: 원자 작업 완료 검증

에이전트 반환 즉시 수행한다. 에이전트가 "모두 완료했다"고 보고해도 이 단계를 건너뛰지 않는다.

1. `현재 세션 작업 목록`로 Phase 3-0에서 등록한 전체 태스크 상태 확인
2. `status`가 `completed`가 아닌 태스크 목록 추출
3. **completed 태스크 대상 파일 실존 확인** — `completed` 태스크 각각의 대상 파일을 Glob 으로 실존 확인하고, 파일이 없으면 자가 보고와 무관하게 **미완료로 간주**하여 2번 목록에 합산한다(현재 세션 작업 목록 갱신은 에이전트 자가 보고라 파일 없는 completed 가 가능하다 — 실존 확인은 도구 1회로 거짓 통과를 결정적으로 차단한다). 수정 태스크는 파일이 원래 존재해 실존만으로는 못 잡으므로, `git status --short` 변경 목록에 해당 경로가 나타나는지로 대조하고 변경 흔적이 없으면 동일하게 미완료로 간주한다(변경이 불필요했다는 에이전트 판단이라면 재호출 반환의 `확인 필요` 로 표면화된다).
4. **미완료 태스크가 있으면**:
   - 해당 파일 + task_id 목록만 추출
   - 동일 에이전트를 재호출 ("아래 파일이 아직 구현되지 않았다. 현재 세션 작업 목록 갱신까지 포함하여 구현하라" 명시)
   - 재호출 후 현재 세션 작업 목록 재확인 (3번 파일 실존 확인 포함)
   - 동일 태스크가 2회 연속 미완료 → stuck 판정 후 에스컬레이션 (Phase 3-2 전용 2회 기준 — 현재 세션 작업 목록은 이진 신호라 부분 진전 개념이 없다. 임계 정본은 `references/error-handling.md` 참조)
5. **모든 태스크 `completed` + 대상 파일 실존 확인 통과** → milestone-tracker 갱신 → Phase 4 진행

**구현 완료 후 milestone-tracker 갱신**

Phase 3-2 통과 후 즉시 `milestone-tracker` 에이전트를 호출하여 해당 항목을 `[x]`로 갱신한다 (🔄 진행 중 처리).

**데이터 흐름**

```
오케스트레이터 → 티켓 파싱 → 현재 세션 작업 목록 (파일별 태스크)
                                    ↓ 태스크 ID 목록
원본 스펙/분석 문서(있을 경우)
                     ↓ 참조
tdd-agent     → 데이터 접근·API 테스트 (Red)
                     ↓ 테스트 파일 목록 + 태스크 ID 목록
백엔드 구현    → 데이터 접근·서비스·API (Green) — 각 파일 완료 시 현재 세션 작업 목록을 completed로 갱신
프론트 구현    → 페이지·컴포넌트 — 각 파일 완료 시 현재 세션 작업 목록을 completed로 갱신
     ↓ 반환 메시지 (오케스트레이터 수신)
오케스트레이터 → 현재 세션 작업 목록 검증 → milestone-tracker 갱신 → Phase 4 진행
```

## Phase 4: 검증

**design-reviewer(UI 마일스톤만)**, **qa-guard**, **security-auditor(조건부)**, **evaluator** 순으로 호출한다.

0. **design-reviewer — 시각적 스펙 검증 (프론트 구현 에이전트가 투입된 경우만)**
   `design-reviewer 에이전트 호출` — 프론트 구현 에이전트 반환 메시지의 `완료 항목` 파일 경로 목록을 입력으로 전달한다. (작업 봉투 `참조`에 `docs/compound/design/README.md` 의무 포함)
   - 산출물: `_workspace/design_review_{scope}_{datetime}.md`
   - 자동 수정 범위: 디자인 토큰 치환, `:hover`/`:focus` 캐스케이드 오버라이드 규칙 추가, `aria-label`/`alt` 속성 추가
   - `구현-에이전트-위임` 항목이 있으면 구현 에이전트 보완 요청 → design-reviewer 재검증을 `approved` 반환 시까지 반복. 동일 FAIL 항목 목록이 3회 연속 변화 없으면 stuck 판정 후 에스컬레이션.
   - **학습 트리거**: 동일 FAIL 항목 목록이 2회 연속 반복되면 stuck 전이라도 `compound-learner` 에이전트를 호출한다(인자: 트리거 `design-repeat`, 보고서 최신 `_workspace/design_review_*.md` 경로). 동일 반복 묶음당 1회만 호출한다.
   - **보완 루프 동안 qa-guard는 실행하지 않는다.** design-reviewer `approved` 반환 후에만 다음 단계 진행.
   - 프론트 구현 에이전트 미투입 마일스톤은 이 단계를 건너뛴다.

1. **qa-guard** — 코드 품질·보안·컨벤션 검증
   `qa-guard 에이전트 호출` — 구현 에이전트 완료 항목의 파일 경로 목록을 입력으로 전달한다. (작업 봉투 `참조`에 `docs/compound/qa/README.md` 의무 포함)
   - 산출물: `_workspace/qa_{feature}_{datetime}.md`
   - **증거 동봉 형식 계약**: tdd-agent 테스트 파일이 존재하는 마일스톤인데 qa 보고서 `## 테스트 결과` 에 실행 명령·러너 출력 인용이 없으면, 보완 루프 판정에 들어가지 않고 qa-guard 에 보고서 재전송을 요구한다(tracker 필수 필드 검증과 동일한 형식 계약 패턴 — 검증자도 LLM이므로 실행 증거 없는 "통과: N"을 통과로 취급하지 않는다). 단 미실행 사유 1줄(예: "테스트 없음 — tdd-agent 미투입", "DB 미연결 — 런타임 검증 보류", "빌드 실패로 보류")이 있으면 형식 충족으로 본다 — 정당한 미실행을 재전송으로 반려하지 않는다. 재전송 시에는 직전 보고서와 **동일한 파일명에 덮어쓰도록** 봉투에 지시한다(새 datetime 파일 금지 — 절대 라운드 상한의 파일 개수 카운트를 부풀리지 않게). 재전송 요구는 stuck 카운터에 합산하지 않는다(검증 내용의 실패가 아니라 형식 결함이므로).
   - 실패 항목 있으면 담당 에이전트에 수정 요청 → qa-guard 재검증을 전 항목 통과 시까지 반복. 동일 실패 항목 목록이 3회 연속 변화 없으면 stuck 판정 후 에스컬레이션.
   - **학습 트리거**: 동일 실패 항목 목록이 2회 연속 반복되면 stuck 전이라도 `compound-learner` 에이전트를 호출한다(인자: 트리거 `qa-repeat`, 보고서 최신 `_workspace/qa_*.md` 경로). 동일 반복 묶음당 1회만 호출한다.

2. **security-auditor — 심층 보안 감사 (조건부)**
   `security-auditor 에이전트 호출` — 구현 파일 목록, qa-guard 보고서, 고위험 매칭 근거를 입력으로 전달한다. (작업 봉투 `참조`에 `docs/compound/security/README.md` 의무 포함 — 카테고리가 있으면)
   - 산출물: `_workspace/security_{scope}_{datetime}.md`
   - 호출 조건은 경로·티켓 문구·qa-guard 반환 신호만으로 판정한다. 코드 의미를 임의 추론해 확대하지 않는다.
     ```text
     if changed_files contain auth/session/permission/admin/payment/webhook/upload/delete
        or personal data read/write
        or server action/API route with high-risk operation:
         run security-auditor
     else if qa-guard returns "보안 심층 검토 필요":
         run security-auditor
     else if user requests release/security audit:
         run security-auditor
     else:
         skip security-auditor
     ```
   - `보안 감사 결과: approved` → evaluator로 진행한다.
   - `보안 감사 결과: 수정필요` → evaluator로 진행하지 않는다. 담당 구현 에이전트 보완 → qa-guard 재검증 → security-auditor 재검증을 반복한다.
   - 동일 보안 위험 목록이 2회 연속 반복되면 `compound-learner` 에이전트를 호출한다(인자: 트리거 `qa-repeat`, 보고서 최신 `_workspace/security_*.md` 경로). 동일 위험 목록이 3회 연속 변화 없으면 stuck 판정 후 에스컬레이션한다.
   - **게이팅 결정 로깅(의무)**: 사용자 보고와 보완 라운드 기록에 `security-auditor 게이팅: 실행 — {근거}` 또는 `security-auditor 게이팅: 스킵 — 고위험 조건 없음`을 1줄 명시한다.
   - **evaluator와의 경계**: security-auditor는 스펙 외 위협 모델을 본다. 원본 스펙·티켓에 명시된 권한 조건의 존재·동등성은 evaluator가 판정한다.

3. **evaluator** — 요구사항 충족 검증
   `evaluator 에이전트 호출` — 마일스톤 번호, 구현 파일 목록, 원본 스펙 문서 경로(있을 경우)를 입력으로 전달한다. (작업 봉투 `참조`에 `docs/compound/evaluation/README.md` 의무 포함)
   - 산출물: `_workspace/eval_{milestone}_{datetime}.md`
   - **증거 동봉 형식 계약**: eval 보고서에 `## 스모크 체크 수행 기록` 섹션이 없거나 비어 있으면(수행 기록도 생략 사유도 없음) evaluator 에 보고서 재전송을 요구한다(qa-guard 와 동일 패턴 — 재전송은 직전 보고서와 동일한 파일명에 덮어쓰게 하고, stuck 카운터에 합산하지 않는다).
   - **의도적 제외 항목 전달 규칙**: 범위에서 뺀 원본 요소가 있으면 evaluator 작업 봉투에 "gap 판정 금지" 같은 침묵 유도 표현을 쓰지 않는다. 대신 "원본에 존재하나 제외된 항목 목록 + 각 승인 출처"를 전달하고 `evaluate` 의 "의도적 제외 목록"에 명시하라고 지시한다. evaluator 는 제외를 통과(`◐`)시키되 보고서 전용 목록에 남긴다 — 승인은 통과를 뜻하지 비가시를 뜻하지 않는다.
   - 누락/초과 항목 있으면 담당 에이전트 보완 요청 → 아래 **증분 재게이팅** → evaluator 재검증을 `approved` 반환 시까지 반복. 동일 미해결 항목 목록이 3회 연속 변화 없으면 stuck 판정 후 에스컬레이션.
   - **증분 재게이팅 — 보완 코드의 상류 게이트 재진입 (의무)**: 담당 에이전트의 보완 반환 `완료 항목`에 있는 신규/변경 파일을 버그 경로 3번과 동일한 동작 레이어 화이트리스트 **경로 prefix 매칭**으로만 분류한다 — "이 코드가 보안에 민감한가" 같은 의미 판단은 금지. 화이트리스트 미정의 프로젝트는 보수적으로 전 파일을 ①로 취급한다(보안 게이트는 graceful skip 대상이 아니다 — 미게이팅보다 과게이팅이 싸다).
     - ① 화이트리스트에 닿는 파일(데이터 접근·서비스·API 핸들러 등) 1개 이상 → `qa-guard` 에이전트를 **해당 파일 한정**으로 증분 재실행한다. 고위험 조건에도 닿으면 security-auditor도 qa-guard 뒤에 재실행한다.
     - ② 닿지 않는 파일(스타일·컴포넌트 등) 1개 이상이고 UI 마일스톤(5-A 와 동일하게 Phase 3 프론트 구현 에이전트 투입 기록으로 판정) → `design-reviewer` 에이전트를 **해당 파일 한정**으로 증분 재검증한다.
     - ①·② 모두 해당하면 design-reviewer → qa-guard → security-auditor(조건부) 순으로 실행하고, 증분 게이트가 전부 통과한 뒤에만 evaluator 재검증으로 진행한다 (Phase 4 게이트 순서 design → qa → security → evaluator 를 증분 재진입에서도 보존).
     - 증분 게이트가 FAIL 을 반환하면 전용 카운터를 새로 만들지 않고 해당 게이트의 기존 보완-재검증 루프에 합류한다 (stuck 임계·동일성 키·상한 규칙은 `references/error-handling.md` 정본 — 무한 재게이팅 차단).
     - **게이팅 결정 로깅(의무)**: 보완 라운드 보고에 `증분 재게이팅: qa-guard {파일 목록} / security-auditor {실행|스킵 및 근거} / design-reviewer {파일 목록} / 해당 없음` 을 1줄 명시한다.
     - **증분 보고서 명명**: 증분 실행의 보고서 파일명은 `_workspace/qa_regate_{feature}_{datetime}.md` / `_workspace/design_review_regate_{scope}_{datetime}.md` 로 봉투에 지정한다(절대 라운드 상한이 게이트 보완 회차의 파일 개수만 세도록 카운트에서 배제 — `references/error-handling.md` 카운트 비대상. 증분 라운드의 유한성은 evaluator 자체 상한이 보장하므로 별도 카운트가 필요 없다).
     - 이유: 게이트 순서가 일방향이라 evaluator 미승인 보완으로 작성된 새 코드는 그대로 두면 evaluator 만 재통과하고 qa-guard(빠른 보안·컨벤션·테스트)·security-auditor(심층 보안)·design-reviewer 를 영영 거치지 않는다 — 새 API 핸들러가 보안 검사 0회로 커밋되는 거짓 통과 채널. 변경 파일 한정 재실행으로 비용을 보완 크기에 상수화한다. (qa-guard 자체 보완 루프의 수정 코드는 qa 를 재통과하므로 대상이 아니다 — 이 규칙의 대상은 본 항의 evaluator 미승인 보완과 5-E 보완이다. 화이트리스트 비매칭 파일이 비-UI 마일스톤에서 보완된 경우는 증분 게이트 비대상으로 남는다 — 의도적 비용 트레이드오프.)
   - **gap이 발견된 경우** (미승인 또는 재검증 통과 불문): 최종 결과 확정 후 `compound-learner` 에이전트를 호출한다(인자: 트리거 `eval-gap`, 보고서 eval 파일 경로)

## Phase 5: 사용자 검증 + 완료 보고

evaluator `approved` 반환 후 마일스톤을 곧바로 ✅ 처리하고 커밋하지 않는다. **프론트 구현 에이전트가 투입된 UI 마일스톤**은 자동 커밋 직전에 사용자 시각/기능 직접 검증 게이트를 반드시 통과시킨다. evaluator 는 코드 차원 정합성을, 사용자는 시각/기능 차원 정합성을 각각 보증한다 — 두 보증이 합쳐져야 진짜 완료다. 흐름은 `5-A → (5-B → 5-C) 또는 5-D` 이며, 부정 응답은 `5-E` 보완 회귀로 빠진다.

> **evaluator 미승인 시 처리**: Phase 4 evaluator 가 미승인 반환했다면 5-A 로 들어오지 않는다. 해당 에이전트 보완 요청 → Phase 4 3번의 **증분 재게이팅** → evaluator 재검증을 `approved` 반환 시까지 반복한다(보완 코드가 상류 게이트를 우회하지 않도록 — 규칙 정본은 Phase 4 3번). approved 후에만 Phase 5 진입.

### 검증 부채 수집 (Phase 5 진입 시 의무)

5-A 판정에 앞서, 이번 마일스톤에서 **검증이 약속보다 약하게 수행된 모든 경우**(= 검증 부채)를 목록으로 수집한다. 닫힌 enum이 아니라 일반 정의이며, 대표 예:

- qa 보고서: "테스트 없음 — tdd-agent 미투입" 표기 후 계속 진행
- qa 보고서: "DB 미연결 — 런타임 검증 보류" (정적 검토만 수행)
- eval 보고서: `## 스모크 체크 수행 기록` 의 생략 사유 (도구 부재 등 — 비-UI 범위의 설계상 생략은 부채가 아니다, 검증 대상 자체가 없으므로)
- 에이전트 1회 재시도 후 재실패로 "해당 결과 없이 진행" 처리된 검증 (`references/error-handling.md` 기본 원칙)
- stuck 에스컬레이션 후 "보류"로 표시된 채 남은 미해결 항목

수집 소스는 최신 `_workspace/qa_*`·`_workspace/security_*`·`_workspace/eval_*` 보고서와 이번 파이프라인의 에러 핸들링 기록이다(보고서 기반 항목은 외부 산출물이라 compaction 후에도 재수집 가능 — 에러 핸들링 기반 항목(재시도 실패·stuck 보류)은 휘발될 수 있으므로 발생 즉시 수집 목록에 기록해 둔다). 이 정의 목록은 여기 1곳이 정본이다 — qa·security-audit·evaluate 스킬은 기존 1줄 표기를 그대로 유지하고 오케스트레이터가 그 표기를 모은다(DRY).

수집한 목록은 두 곳으로 흐른다: ① 5-B step 1 / 5-D step 4 사용자 보고에 재노출, ② 5-C·5-D step 1 milestone-tracker 호출 시 `검증 부채: <목록 | 없음>` 별도 줄로 동봉 → tracker 가 작업 로그(`docs/progress/milestones/M{N}.md`)에 영구 기록. 부채가 있어도 완료를 차단하지 않는다 — "의도적 제외 재노출"과 동일 원리로, 약화된 검증 위에 선 완료 표시를 완전 검증 완료와 산출물 차원에서 구분 가능하게 만든다(승인은 통과를 뜻하지 비가시를 뜻하지 않는다).

### Phase 5-A: 적용 범위 판정

이번 마일스톤 파이프라인에 프론트 구현 에이전트가 한 번이라도 투입됐는지를 본다. 판정은 Phase 3 호출 기록만으로 수행 — 코드 변경 분석 같은 의미 판단은 금지.

- **프론트 구현 에이전트 투입됨** (UI 마일스톤) → 5-B 사용자 검증 게이트
- **프론트 구현 에이전트 미투입** (DB 레이어·결정 문서·마이그레이션 등 헤드리스 마일스톤) → 5-D 즉시 커밋 경로

### Phase 5-B: 사용자 시각/기능 검증 게이트 (UI 마일스톤)

1. **검증 요청 보고서 출력** — 사용자가 무엇을 확인해야 하는지 구체적으로 명시한다. 단순히 "확인 부탁드립니다"는 금지.
   - 구현된 파일 목록
   - qa-guard 결과 요약 (PASS/FAIL 항목)
   - security-auditor 게이팅과 결과 요약 (실행/스킵 근거, 실행 시 approved/수정필요)
   - evaluator 결과 요약 (충족/누락/확인 필요)
   - **원본 대비 의도적 제외 목록 재노출**: evaluator 보고서의 "의도적 제외 목록"이 비어있지 않으면 그 표를 그대로 옮기고 "원본과 달라지는 부분입니다 — 의도와 다르면 알려주세요" 를 덧붙인다. 작업 시작 시점의 추상적 승인을 완료 시점의 구체적 재확인으로 보완한다.
   - **검증 부채 목록 재노출**: "검증 부채 수집"에서 만든 목록이 비어 있지 않으면 그대로 노출하고 "이번 완료 표시는 아래 항목만큼 약한 검증 위에 있습니다 — 직접 확인 시 참고해 주세요" 를 덧붙인다(약화 이벤트를 1줄 보고로 흘려보내지 않고 완료 시점에 묶어 재노출한다).
   - **확인 요청 항목**: 사용자가 브라우저에서 점검할 URL 경로 + 재현 시나리오
   - **응답 게이트 명시**: "이번 변경을 브라우저에서 확인 부탁드립니다. 긍정 응답 후에 milestone-tracker ✅ + 자동 커밋을 진행합니다. 재발 방지 학습(compound-learner) 도 사용자 응답 후 수행됩니다."
2. **사용자 응답 대기** — 이 시점에서는 milestone-tracker ✅ 미갱신, 커밋 미실행, compound-curate 드레인 미발동. 그리고 **재개 프롬프트도 출력하지 않는다** (응답 대기 자체가 next-step).
3. **다음 턴 분기**:
   - **긍정 응답** ("확인됨", "오케이", "잘 됨", "통과" 등) → 5-C 진입
   - **부정/미해결 응답** ("X가 안 됨", "Y가 깨짐", "이 부분 누락" 등) → 5-E 보완 회귀
   - 응답이 모호하면 사용자에게 "긍정 처리해도 될까요, 보완이 필요할까요?" 1회 확인 후 분기

### Phase 5-C: tracker ✅ + 커밋 + 보고 (UI 마일스톤 긍정 경로)

1. **milestone-tracker 최종 갱신** — `milestone-tracker 에이전트 호출` — evaluator의 phase-4-eval 신호를 원형 그대로 전달하고(`검증 결과: approved` 필드를 합성·변형하지 않는다 — tracker 강등 키가 정확 일치 판정이므로 변형 시 정상 완료가 강등된다), `사용자 검증: 통과` 를 별도 줄로 덧붙인다. "검증 부채 수집"의 목록도 `검증 부채: <목록 | 없음>` 별도 줄로 함께 동봉한다(부채가 있으면 tracker 가 작업 로그에 영구 기록. `없음` 도 신호에는 명시하여 수집 단계가 실행됐음을 남긴다 — 원장 기록은 부채가 있을 때만, milestone-track 계약과 동일)
   - 산출물: `docs/progress/milestone-status.md` ✅ 완료 처리
2. **자동 커밋** — `commit` 스킬 사용 (린트 검사 + 커밋)
   - 커밋 메시지 형식: `feat(M{N}): {마일스톤 슬러그 한 줄 요약}`
   - 커밋 실패 시 사용자에게 알리고 수동 커밋을 안내한다
3. **compound-curate 드레인 (커밋 직후)** — Phase 0 "재발 방지 학습 푸시"에서 적재된 `_workspace/curate-needed.md` 를 읽어 비어 있지 않으면, `compound-learner` 에이전트를 백그라운드로 호출하고(인자: 트랙 `compound-curate`, 대상 curate-needed 플래그 목록) **호출 직후 이 파일을 비운다(truncate)** — 비우지 않으면 다음 세션이 이미 처리한 플래그를 재드레인한다. 학습 4트랙과 달리 evaluator `approved` 와 무관하며, 커밋 직후이므로 후속 tracker·commit 호출이 없다. 결과는 4번 보고에 1줄 요약. 파일이 없거나 비어 있으면 이 단계를 건너뛴다.
4. **session-archivist 조건부 호출** — 사용자가 이번 완료에 대해 인계 보존을 요청했거나, 오케스트레이터가 완료 단계에서 persisted handoff가 필요하다고 판단한 경우에만 `session-archivist`를 호출한다. 직접 재개 프롬프트 요청이 아니라 완료 산출물 보존이 목적이다. 산출물은 `docs/history/**` 또는 `docs/handoffs/**`이며 `docs/progress/**`는 수정하지 않는다.
5. **사용자 최종 보고**:
   - milestone-tracker 갱신 결과
   - 자동 커밋 결과 (커밋 해시 포함)
   - compound-curate 드레인 결과 1줄 요약 (드레인이 발동된 경우만)
   - session-archivist 결과 1줄 요약 (호출한 경우만)
   - 다음 스텝 제안
   - **[강제] 다음 스텝 제안 직후 새 세션 재개 프롬프트(단일 ```text 블록)를 반드시 동반 출력한다.**

### Phase 5-D: 비-UI 마일스톤 즉시 커밋 경로 (프론트 구현 에이전트 미투입)

사용자가 브라우저에서 시각 확인할 대상이 없으므로 사용자 검증 게이트를 두지 않는다. 흐름은 아래 step 0 위험 클래스 게이트 통과 후 5-C step 1→2→3→4→5 와 동일하되 검증 요청 보고서(5-B step 1) 는 생략하고 곧바로 진행한다.

0. **위험 클래스 게이트 — 마이그레이션/스키마/백필 (step 1 진입 전 필수).** 완료 항목의 파일 경로·티켓 문구가 마이그레이션·스키마·백필 클래스에 닿는지 **기계 매칭으로만** 판정한다(5-A 와 동일 원칙 — 경로/키워드 매칭만, 코드 의미 판단 금지. 일반 키워드: migration/마이그레이션, schema/스키마, backfill/백필, DDL — 경로 패턴이 프로젝트 AGENTS.md 에 정의돼 있으면 그것을 우선). 안 닿으면 게이트를 건너뛰고 step 1 로 진행한다.
   - **닿음**: dev DB 적용 로그(적용 명령 + 결과 출력 요약)를 증거로 요구한다. qa 보고서 또는 구현 에이전트 반환 메시지에 적용 기록이 있으면 통과 후 step 1 진행(한 번도 dev DB 에 적용되지 않은 스키마 변경이 정적 판정만으로 ✅·커밋되는 것을 막는 게이트 — 적용 실행이 가장 싼 그라운드 트루스다).
   - **적용 로그 부재, 또는 qa 보고서에 "DB 미연결 — 런타임 검증 보류" 표기**: tracker ✅·자동 커밋을 진행하지 않는다. 보류 표기를 재노출하고 사용자에게 "마이그레이션이 dev DB 에 적용 확인되지 않았습니다 — 적용 후 진행할까요, 미적용 상태로 커밋할까요?" 를 1회 확인한 뒤 분기한다(검증 약화를 조용히 통과시키지 않는다). 적용 선택 시 프로젝트 AGENTS.md 가 정의한 적용 명령을 1회 실행해 로그를 확보한 뒤 step 1 로 진행하고(명령 미정의·DB 미연결이면 사용자 수동 적용 결과를 기다린다), 미적용 커밋 선택 시 "마이그레이션 dev DB 미적용 — 사용자 승인" 을 검증 부채 목록에 추가하고(step 1 동봉·step 4 재노출에 포함 — 영구 원장에 남긴다) step 4 보고에 같은 1줄을 명시하고 진행한다. 이 응답 대기 중에는 5-B step 2 와 동일하게 재개 프롬프트를 출력하지 않는다(응답 대기 자체가 next-step).
   - **게이팅 결정 로깅(의무)**: step 4 사용자 보고에 `위험 클래스 게이트: 해당 없음` 또는 `위험 클래스 게이트: 매칭({근거 경로/키워드}) — 증거 {확보 / 부재 → 사용자 분기 결과}` 를 **1줄 명시**한다(버그 경로 evaluator 게이팅 로깅과 동일 원칙 — 게이트 우회를 사후 감사 가능하게).

1. `milestone-tracker 에이전트 호출` ✅ (evaluator approved 신호 원형 그대로 — 사용자 검증 게이트가 없으므로 `사용자 검증: 통과` 줄은 덧붙이지 않는다. `검증 부채: <목록 | 없음>` 별도 줄은 5-C step 1 과 동일하게 동봉한다)
2. `commit` 스킬 사용
3. compound-curate 드레인 (5-C step 3 동일, background)
4. session-archivist 조건부 호출 (5-C step 4 동일)
5. 사용자 보고 (구현된 파일 목록·qa-guard·security-auditor 게이팅·evaluator 결과·**위험 클래스 게이트 판정 1줄(step 0 로깅)**·**의도적 제외 목록 재노출(있을 때)**·**검증 부채 목록 재노출(있을 때)**·tracker 갱신·커밋 해시·드레인 요약·session-archivist 결과(호출 시)·다음 스텝·**[강제] 재개 프롬프트 동반**)

### Phase 5-E: 보완 회귀 (사용자 부정 응답)

1. **응답 본문 분류 — 오케스트레이터 자동 분류.** 사용자 응답에서 결함 성격을 읽고 보완 에이전트를 선택한다.
   - **동작 결함** (특정 기능이 안 됨, 에러 토스트, 데이터 안 보임, 클릭 무반응) → `bug-fixer` 에이전트 호출. 사용자 응답 본문(증상·재현 경로·스크린샷 언급)을 `진단 맥락` 으로 전달.
   - **시각 결함** (스타일 깨짐, 토큰 미반영, hover/focus 잔재, 레이아웃 어긋남) → `bug-fixer` 에이전트 호출 (bug-fixer 가 스타일/마크업도 처리)
   - **마일스톤 스펙 누락 항목** (티켓에 있던 X 가 화면에 없음) → 해당 구현 에이전트에 보완 요청. 작업 봉투 `참조` 에 사용자 응답 본문 첨부.
   - **추가 기능 요청** (티켓 범위 외) → 신규 티켓 분리 권고. 사용자에게 "마일스톤 범위 외 기능으로 보입니다 — 신규 티켓 분리 후 별도 작업이 적절합니다. 현 마일스톤은 이대로 ✅ 처리해도 될까요, 아니면 보류할까요?" 결정 요청.
   - **분류 모호 시** 사용자에게 "어디 계열 보완일까요? [bug-fixer / 프론트 구현 / 백엔드 구현 / evaluator 재검증]" 1회 분류 확인 후 진입.
2. **보완 → 재검증 사이클** — 보완 에이전트 반환 후, 먼저 Phase 4 3번의 **증분 재게이팅** 규칙을 동일 적용한다: 보완 `완료 항목` 파일을 동작 레이어 화이트리스트 prefix 매칭으로 분류해, 닿는 파일은 qa-guard 해당 파일 한정 증분 재실행하고 고위험 조건에 닿으면 security-auditor도 재실행한다. 닿지 않는 파일은 UI 마일스톤이면 design-reviewer 해당 파일 한정 증분 재검증한다. design → qa → security(조건부) → evaluator 순서 보존, 게이팅 결정 1줄 로깅, FAIL 시 해당 게이트 기존 보완-재검증 루프 합류까지 동일하다. 증분 게이트 통과 후 Phase 4 evaluator 재검증 → approved 시 5-A 회귀 (다시 사용자 검증 게이트 진입). (이유: 5-E 보완 코드도 그대로 두면 evaluator 만 재통과하고 qa-guard/security-auditor 보안 검사 0회로 커밋에 도달한다.)
3. **반복 학습 트리거** — 동일 미해결 항목 2회 연속 반복 시 `compound-learner` 에이전트를 호출하되, **트리거는 1번 응답 본문 분류 결과에 대응하는 기존 트랙으로 발행한다**(compound-learner 입력 enum은 `bug-fix | eval-gap | qa-repeat | design-repeat`이며 `user-verify-repeat`는 정의돼 있지 않다 — 격리 호출이라 미정의 트리거의 실패는 표면화되지 않으므로 기존 트랙으로 대응한다). 동작 결함·시각 결함 → `트리거: bug-fix`, 마일스톤 스펙 누락 항목 → `트리거: eval-gap`. 어느 경우든 `보고서: 사용자 응답 본문 + 보완 시도 이력` 을 동봉하고 백그라운드 학습으로 호출한다. 동일 반복 묶음당 1회만 호출한다. (추가 기능 요청은 신규 티켓으로 분리되므로 학습 트리거를 발행하지 않는다.)
4. **stuck 판정** — 동일 미해결 항목 3회 연속 변화 없으면 stuck 판정 → 에스컬레이션.
5. **compound-learner 봉투 주석** — 2회 이상 재보완이 누적된 끝에 사용자 긍정 응답을 받으면, 1차·2차 가설과 정진단 흐름을 compound-learner 봉투에 함께 전달하여 자매 보존 vs `_superseded` 격리 판단을 맡긴다.

## 에러 핸들링

상세 규칙(보완-재검증 루프 전략, stuck 감지, 에스컬레이션 기준)은 `references/error-handling.md`를 읽어 따른다.

**요약**
- 에이전트 실패 시 1회 재시도, 재실패 시 보고서에 명시하고 계속 진행
- 동일 미해결 항목 2회 연속 → compound-learner 호출 (학습)
- 동일 미해결 항목 3회 연속 변화 없음 → stuck 판정 → 에스컬레이션
- 동일 게이트 통산 7회 초과(현 `_workspace` 해당 게이트 보고서 파일 개수 기계 카운트) → 무변화 여부와 무관하게 stuck 판정 → 에스컬레이션

## 테스트 시나리오

정상 흐름·버그 경로·UI 검증·compound-curate·계획 정합성 각 경로의 단계별 드라이런 시나리오는 `references/test-scenarios.md` 에 있다. 워크플로우 정합성을 드라이런으로 검증하거나 특정 경로의 단계 순서를 대조 확인할 때 그 파일을 읽는다. (검증용 참조이며 실행 경로가 아니므로 평상시 파이프라인 진행에는 로드하지 않는다.)
